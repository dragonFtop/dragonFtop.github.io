# UDP 用户数据报协议

UDP：无装饰的传输协议

  • UDP 是一种 “尽力传输的服务”，所以 UDP 传输的 segment 可能会 lost，并且 UDP 传输的 segment 可能会 out of order。
  • connection less：使用 UDP 的 sender 和 receiver 不需要握手建立连接，速度更快，每个 segment 是被独立地处理，互不干扰。

UDP 使用场景：丢包容忍、延迟敏感地场景，比如：视频传输、游戏传输、VoIP

使用 UDP 传输协议的协议：DNS 和 SNMP

# rdt 发展

协议版本面临的具体问题核心解决机制机制详解
rdt1.0无 (基于理想化的可靠信道假设)无 (无需特殊机制)协议假设底层信道是完全可靠的，不会发生比特错误或分组丢失。发送方只需调用 udt_send() 将数据发出，接收方调用 rdt_rcv() 接收并交付给上层即可，双方无需任何交互和确认。
rdt2.0比特差错：数据在传输过程中，比特可能从 0 翻转为 1，或从 1 翻转为 0，导致数据损坏。差错检测 + 反馈重传 (ARQ)1. 校验和 (Checksum)：发送方为数据包计算校验和并附在包中。接收方收到后重新计算，若结果不匹配，则判定数据包已损坏。<br>2. ACK/NAK：接收方根据校验结果，向发送方反馈。若数据正确，发送 ACK (确认)；若数据损坏，发送 NAK (否认)，要求重传。<br>3. 重传：发送方收到 NAK 后，会重新发送导致 NAK 的那个数据包。
rdt2.1ACK/NAK 损坏：接收方返回的 ACK 或 NAK 反馈信息本身在传输中也可能损坏。发送方收到损坏的反馈后，无法判断接收方的真实状态。引入序号 (Sequence Number)1. 序号：发送方为每个数据包附加一个序号（如 0 和 1 交替使用）。<br>2. 处理重复：当 ACK/NAK 损坏时，发送方会选择重传。接收方通过检查数据包的序号，可以判断这是期待的新数据包还是一个因重传导致的重复数据包。若是重复包，则直接丢弃，避免向上层交付重复数据。
rdt2.2机制冗余：rdt2.1 使用 ACK 和 NAK 两种反馈，功能上存在冗余，可以简化。仅使用 ACK1. 取消 NAK：协议只使用 ACK 进行反馈。<br>2. 重复 ACK 即 NAK：接收方在收到损坏的数据包时，不再发送 NAK，而是重复发送上一个正确接收数据包的 ACK。例如，接收方已收到 P0，但 P1 损坏，它会再次发送 ACK 0。发送方收到这个重复的 ACK 0，就明白 P1 传输失败，从而重传 P1。这相当于用重复的 ACK 隐式地表达了 NAK 的功能。
rdt3.0分组丢失：数据包或 ACK 可能在网络中彻底丢失，而不仅仅是损坏。此时，发送方会一直等待一个永远不会到来的 ACK，导致 “死锁”。超时重传 (Timeout)1. 定时器 (Timer)：发送方在发出一个数据包后会启动一个定时器。<br>2. 超时重传：如果在定时器超时前仍未收到对应的 ACK，发送方就认为该数据包或 ACK 已丢失，并自动重传该数据包。序号机制同样用于处理因超时重传可能产生的重复包。

# rdt3.0 的性能评估

性能评估主要通过计算发送方利用率 Sender Utilization 来进行，它衡量的是发送方实际用于发送数据的时间占总时间的比例。

📉 性能瓶颈：停等协议

rdt3.0 是一个典型的停等协议 Stop-and-Wait Protocol 。这意味着发送方每发送一个数据包后，必须停止发送并等待接收方的确认（ACK），才能发送下一个数据包。在等待期间，宝贵的信道带宽被完全闲置，造成了巨大的资源浪费。

🧮 利用率计算实例

我们可以通过一个经典例子来量化评估 rdt3.0 的性能：

场景假设
链路带宽 (R): 1 Gbps (即 10⁹ bps)
传播时延 (RTT/2): 15 ms (例如，横跨大陆的距离)
数据包大小 (L): 1000 字节 = 8000 比特

计算步骤
发送时延 (Transmission Delay):
将一个数据包的所有比特推到链路上所需的时间。
发送时延 = L / R = 8000 比特 / 10⁹ bps = 8 微秒 (μs)

往返时间 (RTT):
发送方发出数据到收到确认的总时间。
RTT = 2 × 传播时延 = 2 × 15 ms = 30 毫秒 (ms) = 30,000 微秒 (μs)

发送方利用率 (U_sender):
发送方实际忙于发送的时间与总耗时（发送时延 + RTT）的比值。
U_sender = 发送时延 / (发送时延 + RTT)
U_sender = 8 μs / (8 μs + 30,000 μs) ≈ 0.00027

📌 结论

计算结果表明，发送方的利用率仅为 0.027%。这意味着在 99.97% 的时间里，发送方和信道都处于空闲等待状态。

# pipelined protocol

解决 rdt3.0 的性能瓶颈:pipelined protocol，允许发送方在发送一个数据包后立即开始发送下一个数据包，而不是等待确认。

# Go-Back-N protocol

引入 pipelined protocol 存在的问题：一旦出现问题，如何让接收方最终能够接收到完整数据是比较困难的，所以引入了 Go-Back-N protocol。GBN 中的 sender 的 windows 宽度为 N，receiver 的 windows 宽度为 1。sender 的窗口 base 停在发送了但没有受到 Ack 的序列，receiver 的窗口 base 停在没有受到数据包的序列。若发生重传，sender 方会重新发送更新的 base 直到之前的 base+N-1 的序列，也就是前一次发送但没有受到 ack 的序列以及之后的序列。

# TCP 报文

TCP 的报文结构：

TCP 中的 Seq 值是指每个数据包段中第一字节数据在字节流中的编号
在 TCP 中 receiver 返回的 ack 的值是下一次期望接收的字节编号
TCP 是累计确认 (cumulative ack) 和捎带确认 (Piggyback)

# TCP 中的 RTT 和 Timeout

实际情况中 TCP 的 RTT 是动态变化的，但是需要知道 RTT 来设计 Timeout 的 Interval，所以这里我们需要去估计 RTT（Estimate RTT）, 如果估计的 RTT 过短，会造成 prematrue 的 timeout 发生，同时造成不必要的重传；如果估计的 RTT 过长，会造成对报文丢失的够慢响应（重传响应过慢）。

计算 Timeout Interval 的过程：

  1. 知道 sampleRTT，measured time from segment transmission until ACK receipt。sampleRTT 是动态变化的。
  2. 计算 EstimatedRTT，EstimatedRTT = (1 - a) * EstimatedRTT + a * sampleRTT
  3. 计算 RTT 安全边界（safe margin）DevRTT，DevRTT = (1 - b) * DevRTT + b * |sampleRTT - EstimatedRTT|
  4. 计算 Timeout Interval，TimeoutInterval = EstimatedRTT + 4 * DevRTT

# TCP 中的 RDT 机制

TCP 的 RDT 实现在 IP 协议的不可靠服务的基础之上。

  • pipelined segment, 流水线报文
  • cumulative ack, 累计确认
  • single retransmission timer, 单个重传计时器。
    TCP 中重传的触发因子：
  • Timeout event，报文传输超时
  • Duplicate ACK，收到重复的 ACK

# TCP 中的事件

  1. sender 方的 events：
    • 创建带有序号的 segment（序号是 segment 中第一个字节在字节流中的编号）
    • 如果 timer 没有启动，启动 timer
    • Timeout，重传引起 Timeout 的 segment，重新启动 timer
    • sender 方收到 Ack 后要先检查这个 Ack 是否是确认的是未被确认的 segment（确认是否为 duplicate ACK）
    • 确认是有效 ack 后，更新滑动窗口等状态
    • 如果还有未确认的 segment，Timer 重新启动
  2. receiver 的 events：
接收方事件TCP 接收方动作
收到具有预期序列号且按序到达的报文段。<br> 直到该预期序列号的所有数据都已被确认延迟确认。<br> 等待下一个报文段最多 500ms。如果没有下一个报文段，则发送确认 (ACK)
收到具有预期序列号且按序到达的报文段。<br> 另有一个报文段正在等待确认立即发送单个累积确认，<br> 同时确认这两个按序到达的报文段
收到序列号高于预期的乱序报文段。<br> 检测到缺口立即发送重复确认，<br> 指示下一个预期字节的序列号
收到的报文段部分或完全填补了缺口立即发送确认，前提是该报文段从缺口的低端开始

# TCP 中的 Flow Control

TCP 连接的 receiver 方有 receive buffer (接收缓存)。
TCP 的 Flow Control：
sender 方传输速度适中，不会使 receiver 方接收缓存溢出。
Speed-matching service：send rate 与 receive rate 相契合。
总得来说，Flow Control 的实现约等于是 receiver 控制 sender 的发送速率。

# TCP 中的连接控制

  1. 创建连接：三次握手
    • 第一步：客户端主机向服务器发送 TCP SYN 报文段
      • 指定初始序列号 (initial seq #)
      • 无数据，但是要消耗一个序号
    • 第二步：服务器主机收到 SYN，回复 SYNACK 报文段
      • 服务器分配缓冲区
      • 指定服务器初始序列号 (initial seq. #)
      • 无数据，消耗一个序号
    • 第三步：客户端收到 SYNACK，回复 ACK 报文段，该报文段可包含数据
  2. 关闭连接，四次挥手
    • 第一步：客户端发起关闭
      • 动作：客户端调用 close ()，发送 FIN 报文（FINbit=1, seq=x）。
      • 含义：告诉服务器 “我没数据要发了”。
      • 状态：客户端进入 FIN_WAIT_1。
    • 第二步：服务器确认收到
      • 动作：服务器收到 FIN，回复 ACK 报文（ACKbit=1, ACKnum=x+1）。
      • 含义：告诉客户端 “我知道你不想发了”。
      • 状态：客户端进入 FIN_WAIT_2；服务器进入 CLOSE_WAIT（此时服务器还可以继续发数据）。
    • 第三步：服务器发起关闭
      • 动作：服务器发完剩余数据后，发送 FIN 报文（FINbit=1, seq=y）。
      • 含义：告诉客户端 “我也发完了，可以断了”。
      • 状态：服务器进入 LAST_ACK。
    • 第四步：客户端确认并等待
      • 动作：客户端回复 ACK 报文（ACKbit=1, ACKnum=y+1）。
      • 含义：确认服务器的关闭请求。
      • 状态：服务器进入 CLOSED（连接彻底关闭）；客户端进入 TIMED_WAIT（等待 2 倍最大报文寿命以确保服务器收到 ACK），随后进入 CLOSED。

# TCP 中的拥塞控制 (Congestion Control)

拥塞控制中的重点概念：带宽检测、拥塞窗口、接收窗口、慢启动、拥塞避免、三次重复确认、超时、快速重传、快速恢复。

  1. 带宽检测 (probing for bandwidthc)：指发送数据的速率持续增大，当遇到数据包 loss 时时候，说明网络拥塞，带宽不足，减小发送数据的速率的这类方式。
  2. 拥塞窗口 (congestion window)：sender 方发送数据包的窗口，一般度量为多少个 MSS (最大段大小)，拥塞窗口是动态变化的，且存在阈值，其值取决于 min (cwnd, rwnd)，其实还需要考虑 ssthresh（阈值）。
  3. 接收窗口 (receive window)：receiver 方接收数据包的窗口，其大小为接收 buffer 的剩余空间。
  4. 慢启动 (slow start)：拥塞窗口指数级 (翻倍) 增长，便于快速探测网络中的最大可用带宽，尽可能快速传输。
  5. 拥塞避免 (congestion avoidance)：当 cwnd>=ssthresh 时，拥塞窗口线性（+1）增长，以逐步适应网络中的真实带宽。
  6. 三次重复确认 (three-way handshake)：如果某一个 segment 发送后收到三个 ack 的序号 == 该 segment 的 seq，则认为该 segment 丢失，并重新发送该 segment。
  7. 超时 (timeout)：当某一个 segment 发送后没有在 timeout interval 内收到 ack，则认为该 segment 丢失，并重新发送该 segment。
  8. 快速重传 (fast retransmit)：由于三次重复确认而被判为丢包的情况下的数据包快速重传机制。
  9. 快速恢复 (fast recovery)：与快速重传机制同时进行，拥塞窗口大小减半，并进入拥塞避免模式。
