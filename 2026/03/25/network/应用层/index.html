# CS 模式 (client-server)

区分服务器（sever）和客户端 (client):

  1. server：
    always-on host (总是开机的主机)
    permanent IP address (固定 IP)
    Data center for scaling (数据中心)
  2. client:
    communicate with server（和服务通信）
    may be intermittently connected（间断连接）
    may have dynamic IP（可能是动态 IP）
    do not communicate directly with each other（不与其他客户端直接通信）

# P2P 模式

在 P2P 模式中，没有固定的服务器和客户端区分：

  1. no always-on server（没有常开的服务器）
  2. arbitrary end systems directly communicate（主机直接通信）
  3. peers request service from other peers, provide service in return to other peers（两台主机间彼此提供服务）
  4. peers are intermittently connected and change IP addresses（两台主机是间断连接，IP 地址可能动态改变）

# 可供应用程序使用的运输服务

应该为应用程序提供神那么样地传输服务主要取决于应用程序的需求，具体需求可以分为以下四种：

  1. 数据传输是否可靠：
    可靠数据传输：数据能够完整无差错地从源发送到目的地接收。
    不可靠数据传输：数据能够从源发送到目的地接收，但可能丢失或者错误。
  2. 吞吐量：
    带宽敏感应用：需要维持足够的带宽，以保证传输结果的稳定。比如：视频传输、文件传输、游戏传输，需要实时稳定的传输才能使效果好。
    弹性应用：不需要维持固定的带宽，比如：聊天、新闻、邮件，不需要实时的传输。
  3. 定时：
    有的应用要求低延迟（low delay），比如游戏应用。
    某些应用允许较高延迟（high delay），比如聊天应用。
  4. 安全：
    传输过程需要数据加密等安全操作。

# Web 和 HTTP

Web 的应用协议是超文本传输协议 HTTP (HyperText Transfer Protocol)

  1. 持续 HTTP 连接 (persistent connection)
    Use TCP protocol to maintain connection.
    （1）client 通过 TCP 协议建立连接
    （2）server 接受 client 发起的链接请求
    （3）client 的 browser 向 server 发送 HTTP 请求报文
    （4）server 返回 HTTP 响应报文
    （5）TCP 连接 close
    在这种持续的连接中，server 不会记录 client 过去的 request 内容。
  2. 非持续 HTTP 连接（non-persistent connection）
    与持续的 http 连接不同，非持续 http 连接中，client 与 server 之间单次 TCP 连接仅传输一个文件，完成单次传输后关闭 TCP 连接，传输下一个文件时创建新的 TCP 连接，重复直到请求与响应完成。

# POP3 协议

POP3 ：Post Office Protocol，3.0，邮局协议 3.0。
POP3 is stateless across sessions，but the server will maintain states。也就是说，POP3 协议是无状态的，但是服务器会保存状态。所以其是偏向于无状态的。

# IMAP 协议

IMAP ：Internet Mail Access Protocol，3.0，互联网邮件访问协议 3.0，是有状态的协议。

它在应用层实现，一般存在 application-layer protocol host, routers, name servers 这些组件来实现名字与 IP 地址的映射。

# P2P 应用

P2P 应用：Peer-to-Peer，即点对点应用。
特点：

  1. 无中心化：P2P 应用没有中心服务器，所有的数据都存储在各个节点上。no always-on server
  2. 两个终端之间直接通信。end sysstems directly communicate with each other
  3. 终端之间建立间歇性的连接，同时会动态改变 IP 地址。peers are intermittently connected aand change IP addresses

C-S 模型和 P2P 模型文件传输时间比较：

  1. C-S 模型：

    解释： NF/us 服务器上传 N 份 F 文件的时间， F/dmin 下行速度最慢的客户端下载 F 文件的时间。总传输时间取决于上述二者瓶颈（最大值）。
  2. P2P 模型：

    解释： F/us 服务器端上传一次 F 文件的时间， F/dmin 下行带宽最小的客户端下载 F 文件的时间， NF/(us+sigma(ui)) 全部参与 F 文件上行的终端所需要的时间。总传输时间取决于上述三者的最大值。

# P2P 索引

P2P 应用中，客户端需要获取目标文件所在的节点的位置信息，也就是需要索引来告诉客户端，目标文件所在的节点。根据这一过程可分为两类：中心化索引 (centralized indexing) 和分布式索引 (distributed indexing)。

  1. 中心化索引：
    中心化索引方式中存在一个 index server，index server 存储了所有文件的位置信息，也就是文件所在目标主机的 IP+port，client 通过 index server 获取文件位置信息。
    例子：Napster：
    peer 上线并向索引服务器报备（资源信息和位置信息）
    peer 向索引服务器请求文件位置信息
    index server 返回文件位置信息
    peer 之间直接建立连接，进行文件传输
    传输完成，peer 下线

缺点：存在单点故障问题，如果 idnex server 挂掉，则无法获取文件位置信息。索引服务器存在带宽问题，也就是瓶颈。
2. 分布式索引：
覆盖网络 (overlay network)：对于 P2P 场景中的 peer 与 peer 之间连接关系的拓扑结果。
查询洪泛 (querry flooding)：peer A 向自己所在的覆盖网络中的邻居发起查询请求，被请求的邻居如果存在该文件，则返回自己的索引给 peer A 来连接传输，否则邻居将此请求转发给自己的邻居。

  1. 层次化覆盖网络
    介于中心化和分布式索引之间，存在层次化覆盖网络。
    在层次化覆盖网络中存在超级节点 (super node) 和其归属下的一般节点 (ordinary peer)。超级节点充当索服务器的功能。

# 应用层

# 一、网络应用原理

# （一）网络应用体系结构

# 1. C/S 模式（Client-Server）

核心特征：服务器提供服务，客户端主动请求，集中式服务模型。

  • 服务器（Server）
    • 永久在线（always-on host）、固定 IP 地址、7×24 小时运行
    • 性能强大，常采用数据中心集群扩展，处理海量并发
    • 被动响应，不主动发起通信
  • 客户端（Client）
    • 主动与服务器通信，间歇性连接
    • 动态 IP（DHCP 分配），不直接互通
    • 资源有限，依赖服务器提供数据与计算

# 2. P2P 模式（Peer-to-Peer）

核心特征：无固定服务器，端到端直接通信，分布式自组织。

  • 无永久在线服务器，任意端系统直接通信
  • 节点既是客户端（请求服务）也是服务器（提供服务）
  • 节点动态加入 / 退出、IP 常变、间歇性连接
  • 扩展性强：节点越多，整体服务能力越强

C/S vs P2P 对比（考研高频）

特性C/SP2P
中心化强中心化（单点瓶颈）完全分布式（无中心）
可靠性服务器故障则服务瘫痪部分节点失效不影响整体
带宽成本服务器上行带宽为瓶颈节点分担上传，总带宽高
部署成本高（服务器 + 运维）低（利用用户节点资源）
典型应用Web、邮件、数据库BT、迅雷、直播、区块链

# （二）进程通信

# 1. 进程与通信模型

  • 进程（Process）：主机中运行的应用程序实例，网络通信基本单元
  • 客户进程：主动发起通信的进程
  • 服务器进程：被动等待连接、响应请求的进程
  • 通信前提：跨网络进程必须通过传输层协议（TCP/UDP）交换报文

# 2. 套接字（Socket）—— 应用编程接口（API）

  • 定义：应用层与传输层之间的编程接口，是进程收发数据的门户
  • 本质：对 TCP/IP 的封装，提供统一调用接口
  • 标识： Socket = (IP地址, 端口号) ，唯一标识主机上一个进程
  • 端口号：16 位整数（0~65535）
    • 熟知端口（0~1023）：HTTP(80)、HTTPS(443)、FTP(21)、DNS(53)、SMTP(25)、POP3(110)
    • 注册端口（1024~49151）：应用程序专用
    • 动态端口（49152~65535）：客户端临时使用

# 3. 进程寻址（完整地址）

  • 主机标识：32 位 IPv4 地址（唯一标识主机）
  • 进程标识：16 位端口号（唯一标识主机内进程）
  • 完整地址： IP地址:端口号 （如 192.168.1.100:8080 ）

# （三）应用程序的运输服务需求（4 大核心）

  1. 可靠数据传输
    • 可靠：无丢失、无差错、按序（文件、网页、邮件）
    • 不可靠：允许丢包 / 乱序（实时音视频、游戏）
  2. 吞吐量（带宽）
    • 带宽敏感：需稳定带宽（视频、文件传输）
    • 弹性：自适应可用带宽（聊天、邮件、网页）
  3. 定时（延迟）
    • 低延迟要求：游戏、视频通话（<150ms）
    • 高延迟容忍：邮件、文件下载
  4. 安全性
    • 机密性（加密）、完整性（防篡改）、身份认证

# （四）因特网提供的运输服务（TCP/UDP）

# 1. TCP（传输控制协议）

  • 面向连接：三次握手建连、四次挥手断连
  • 可靠传输：确认应答、超时重传、序列号、校验和
  • 流量控制：滑动窗口（防止接收方溢出）
  • 拥塞控制：慢启动、拥塞避免、快重传、快恢复
  • 字节流：无边界，应用层需自定义分包
  • 适用：HTTP、HTTPS、FTP、SMTP、POP3、远程登录

# 2. UDP（用户数据报协议）

  • 无连接：直接发数据报，不握手、不维护状态
  • 不可靠：无确认、无重传、可能丢包 / 乱序
  • 头部小：仅 8 字节（TCP≥20 字节），开销低
  • 无拥塞 / 流量控制：发送速率由应用决定
  • 数据报：每个报文独立、有边界
  • 适用：DNS、视频直播、游戏、VoIP、实时监控

TCP vs UDP 核心对比（必背）

特性TCPUDP
连接性面向连接（虚电路）无连接（数据报）
可靠性可靠（不丢、不乱、不重）不可靠（可能丢包）
传输单位字节流独立数据报
头部开销20~60 字节8 字节
流量 / 拥塞控制有无
传输效率低（延迟高、开销大）高（延迟低、速度快）
应用场景可靠优先（网页、文件、邮件）实时优先（音视频、DNS）

# 二、Web 和 HTTP

# （一）HTTP 基础

  • 全称：HyperText Transfer Protocol（超文本传输协议）
  • 定位：Web 应用层协议、C/S 模型、请求 - 响应模式
  • 特点
    • 无状态（Stateless）：服务器不保存客户端历史状态
    • 灵活可扩展：支持多种 MIME 类型（HTML、图片、JSON）
    • 简单快速：报文文本格式，易调试
  • 默认端口：80（HTTP）、443（HTTPS）

# （二）HTTP 连接方式

# 1. 非持久连接（HTTP/1.0 默认）

  • 机制：一个 TCP 连接仅传输一个对象，传输完立即关闭
  • RTT 开销：请求一个对象 = 1 个 RTT（建连）+1 个 RTT（请求 / 响应）= 2RTT / 对象
  • 缺点：连接建立开销大、并发低、服务器负载高

# 2. 持久连接（HTTP/1.1 默认，Keep-Alive）

  • 机制：TCP 连接保持，同一客户端 / 服务器复用连接传输多个对象
  • 分类
    • 非流水线（默认）：串行请求，前一个响应完再发下一个
    • 流水线（Pipeline）：并行发送多个请求，减少等待
  • 优点：降低 RTT、减少 TCP 开销、提升效率

# （三）HTTP 报文格式

# 1. 请求报文（客户端→服务器）

  • 请求行三要素
    • 方法（Method）：GET（查）、POST（增）、PUT（改）、DELETE（删）、HEAD（仅响应头）、OPTIONS（查支持方法）
    • URL：资源路径
    • 版本：HTTP/1.0、HTTP/1.1、HTTP/2、HTTP/3
  • 常见请求头：Host、User-Agent、Referer、Cookie、Content-Type

# 2. 响应报文（服务器→客户端）

  • 状态码（5 类，必背）
    • 1xx（信息）：100 Continue（继续发体）、101 Switching Protocols
    • 2xx（成功）：200 OK（正常）、201 Created（创建）、204 No Content（无体）
    • 3xx（重定向）：301 Moved Permanently（永久）、302 Found（临时）、304 Not Modified（缓存）
    • 4xx（客户端错）：400 Bad Request、403 Forbidden、404 Not Found
    • 5xx（服务器错）：500 Internal Error、502 Bad Gateway、503 Service Unavailable

# （四）HTTP 状态管理（Cookie/Session）

  • Cookie（客户端存储）
    • 服务器通过 Set-Cookie 响应头发送，浏览器本地保存
    • 下次请求自动带 Cookie 头，维持会话状态（登录、购物车）
  • Session（服务器存储）
    • 服务器为每个客户端创建 Session，存用户状态
    • 用唯一 SessionID 关联（通常存在 Cookie）
    • 更安全，敏感数据不存客户端

# （五）Web 缓存（Proxy）

  • 作用：本地缓存副本，减少服务器负载、降低延迟、节省带宽
  • 校验机制： If-Modified-Since / If-None-Match
  • 命中：返回 304 Not Modified，直接用缓存
  • 未命中：返回 200 + 新内容，更新缓存

# （六）HTTPS（HTTP Secure）

  • 原理： HTTP + TLS/SSL （加密层）
  • 端口：443
  • 功能
    • 机密性：对称加密传输数据
    • 完整性：哈希校验防篡改
    • 身份认证：数字证书验证服务器身份
  • 流程：TCP 三次握手→TLS 握手（协商加密算法、交换密钥）→加密传输 HTTP

# 三、域名系统 DNS

# （一）DNS 基础

  • 全称：Domain Name System（域名系统）
  • 核心功能：域名↔IP 地址映射（分布式层次数据库）
  • 协议：应用层协议，默认 UDP 53（区域传输用 TCP 53）
  • 域名结构：层次树状（根→顶级域→二级域→主机）
    • 根域（.）→顶级域（com、cn、org、edu）→二级域（baidu.com、sina.com.cn）→三级域（www.baidu.com）

# （二）DNS 服务器分级（4 类）

  1. 根域名服务器（Root）
    • 全球 13 个（A~M），返回顶级域服务器 IP
  2. 顶级域服务器（TLD）
    • 管理对应顶级域（.com/.cn），返回权威服务器 IP
  3. 权威域名服务器（Authoritative）
    • 管理具体域名（baidu.com），最终返回 IP 记录
  4. 本地域名服务器（Local / 递归）
    • ISP / 内网提供，客户端默认 DNS，代理查询、缓存结果

# （三）DNS 解析过程（递归 + 迭代，高频）

  1. 客户端查浏览器 / OS 缓存→有则直接返回
  2. 无则发查询给本地 DNS 服务器
  3. 本地 DNS→根服务器：查顶级域（.com）IP
  4. 本地 DNS→顶级域服务器：查权威域（baidu.com）IP
  5. 本地 DNS→权威服务器：查主机（www.baidu.com）IP
  6. 本地 DNS 缓存结果，返回 IP 给客户端

查询方式

  • 递归查询（客户端→本地 DNS）：本地 DNS 代查到底，返回最终结果
  • 迭代查询（本地 DNS→根→TLD→权威）：每级返回下一级地址，本地逐级查

# （四）DNS 资源记录（RR）

  • A 记录：域名→IPv4（最常用）
  • AAAA 记录：域名→IPv6
  • NS 记录：域→权威 DNS 服务器域名
  • CNAME 记录：别名→规范域名
  • MX 记录：邮件服务器域名
  • TTL：缓存生存时间（秒）

# 四、文件传输协议 FTP

# （一）FTP 基础

  • 全称：File Transfer Protocol（文件传输协议）
  • 模式：C/S、双 TCP 连接（控制 + 数据）
  • 端口：控制连接 21（持久）、数据连接 20（主动模式）

# （二）连接机制（带外控制）

  1. 控制连接：全程保持，传命令 / 响应（PORT、PASS、LIST、RETR）
  2. 数据连接：传输文件 / 目录列表，传完关闭
  • 主动模式（Port）：服务器主动连客户端 20 端口（NAT 环境有问题）
  • 被动模式（Pasv）：客户端连服务器随机端口（穿透 NAT，常用）

# （三）FTP 工作流程

  1. 客户端→服务器：TCP 建连（21 端口）
  2. 认证（USER/PASS）
  3. 命令交互（CWD、LIST、RETR、STOR）
  4. 数据连接传输文件
  5. 退出（QUIT），关闭控制连接

# 五、电子邮件（SMTP/POP3/IMAP）

# （一）系统组成

  • 用户代理（UA）：Outlook、Foxmail、网页邮箱（写 / 读 / 管理）
  • 邮件服务器：发送（SMTP）、接收（存储）
  • 协议
    • 发送：SMTP（25 端口）
    • 接收：POP3（110）、IMAP（143）

# （二）SMTP（发送）

  • C/S、TCP 25、持久连接
  • 流程：建连→HELO→AUTH→MAIL FROM→RCPT TO→DATA（正文）→QUIT
  • 限制：仅传 ASCII，非文本用 MIME 编码（图片 / 附件）

# （三）POP3 vs IMAP（接收）

  • POP3（邮局协议）
    • 下载到本地，服务器删除（或保留副本）
    • 简单、不支持服务器端管理
  • IMAP（网际消息访问协议）
    • 在线操作服务器邮箱，本地仅缓存
    • 支持文件夹、未读标记、多设备同步

# 六、P2P 应用

# （一）P2P 核心特点

  • 无中心化：无专用服务器，节点对等
  • 直接通信：端到端数据传输，不经过中心
  • 动态性：节点自由进出、IP 动态变化
  • 自组织：节点自发组网、协同服务
  • 可扩展性：节点越多，总带宽 / 能力越强

# （二）C/S vs P2P 文件分发时间（计算高频）

参数定义

  • F ：文件大小（bit）
  • us ：服务器上行带宽（bps）
  • di ：第 i 个客户端下行带宽（bps）
  • ui ：第 i 个客户端上行带宽（bps）
  • N ：客户端总数
  • dmin = min(d1,d2,...,dN) （最慢客户端）
  • U_total = us + u1+u2+...+uN （总上行）

# 1. C/S 模型时间

D_CS = max(NF/us, F/dmin)

  • 瓶颈 1：服务器上传 N 份（ NF/us ）
  • 瓶颈 2：最慢客户端下载（ F/dmin ）
  • 结论：N 越大， NF/us 越大，服务器为瓶颈

# 2. P2P 模型时间

D_P2P = max(F/us, F/dmin, NF/U_total)

  • 瓶颈 1：服务器至少传 1 次（ F/us ）
  • 瓶颈 2：最慢客户端（ F/dmin ）
  • 瓶颈 3：总上行分发 N 份（ NF/(us+Σui) ）
  • 结论：N 越大， NF/U_total 越小，P2P 优势越明显

# （三）P2P 索引机制（资源定位）

# 1. 中心化索引（Napster）

  • 中心索引服务器：存所有节点资源 + IP:Port
  • 流程：节点注册→查询→服务器返回位置→P2P 直传
  • 缺点：单点故障、带宽瓶颈、易监管

# 2. 分布式索引（Gnutella）

  • 覆盖网络（Overlay）：节点逻辑拓扑（物理网之上）
  • 查询洪泛（Flooding）：节点向邻居广播查询，逐层转发，命中返回
  • 优点：完全分布式、无单点、抗毁强
  • 缺点：洪泛开销大、效率低、延迟高

# 3. 层次化覆盖（混合式，BitTorrent）

  • 超级节点（Super Node）：高性能节点，充当区域索引
  • 普通节点：连超级节点，查询先查超级节点
  • 平衡：效率 + 分布式，兼顾性能与健壮性

# 七、动态主机配置协议 DHCP

# （一）DHCP 基础

  • 全称：Dynamic Host Configuration Protocol
  • 功能：自动分配 IP、子网掩码、网关、DNS、租期
  • 端口：服务器 67、客户端 68（UDP）
  • C/S 模型、即插即用

# （二）DHCP 工作流程（DORA）

  1. Discover（发现）：客户端广播（0.0.0.0:68→255.255.255.255:67）找 DHCP 服务器
  2. Offer（提供）：服务器单播回应， offer 临时 IP
  3. Request（请求）：客户端广播，选中某服务器 IP
  4. Acknowledge（确认）：服务器正式分配 IP + 参数 + 租期

# （三）租期与续约

  • 租期：IP 有效时间（如 1 天）
  • 50% 租期：客户端单播发续约请求
  • 87.5% 租期：仍未成功则广播重找
