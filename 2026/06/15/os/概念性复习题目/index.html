# 第七章 内存管理 概念性复习题

# 7.1 内存管理需要满足哪些需求？

内存管理需要满足四大需求：

需求说明
内存空间的分配与回收为进程分配所需的内存空间，进程结束后回收
内存空间的扩充通过虚拟内存技术从逻辑上扩充内存容量
地址转换将程序中的逻辑地址转换为内存中的物理地址
存储保护确保每个进程只能访问自己的内存区域

存储保护的具体实现方式包括：设置上下限寄存器，或采用重定位寄存器（基址寄存器）+ 界地址寄存器（限长寄存器）的组合。

# 7.2 为何需要重定位进程的能力？

程序在编译时并不知道自己会被加载到内存的哪个位置运行，这取决于运行时的内存使用情况。多道程序环境下，进程可能被换出后再换入到不同的内存区域。因此操作系统必须能够将程序中的逻辑地址重定位为实际的物理地址，进程才能正确执行。

三种重定位（装入）方式：绝对装入、静态重定位（装入时转换）、动态重定位（运行时转换）。

# 7.3 为何不可能在编译时实施内存保护？

编译时无法预知程序实际运行时内存中其他进程的存在和布局。内存保护需要在运行时由硬件和操作系统共同完成 —— 检查每次内存访问的地址是否在合法范围内（越界则触发中断）。这些信息只有到了运行时刻才能确定，编译期无从得知。

# 7.4 允许两个或多个进程访问内存某一特定区域的原因是什么？

主要有两个场景：

  1. 进程间通信（IPC） ：多个进程通过共享内存区域来交换数据，高效且无需内核中转。
  2. 共享代码段：多个进程执行同一程序（如多个 shell 实例）时，可共享同一份只读代码段（纯代码 / 可重入代码），节省内存。纯代码是不可修改的代码，不属于临界资源，可被多个进程安全地并发访问。

# 7.5 在固定分区方案中，使用大小不等的分区有何好处？

可以更灵活地容纳不同大小的进程：小进程分配到小分区，大进程分配到大分区，减少因分区大小固定而导致的内部碎片。相比大小均匀的分区方案，不等分区更能适应实际工作负载中进程大小参差不齐的情况。

# 7.6 内部碎片和外部碎片有何区别？

碎片类型定义产生场景
内部碎片分配给某进程的内存区域中，未被该进程实际使用的部分固定分区 / 分页，进程大小小于分区或页面容量的整数倍
外部碎片内存中空闲分区总和满足需求，但每个单独的空闲分区都太小而无法利用动态分区分配，频繁的分配和释放后留下的细小空隙

外部碎片可通过紧凑技术（移动已分配分区使碎片合并）来解决。分页存储管理则从根本上避免了外部碎片（但仍有少量内部碎片）。

# 7.7 逻辑地址、相对地址和物理地址有何区别？

地址类型定义视角
逻辑地址 / 相对地址程序视角的地址，从程序起始位置开始的偏移量用户 / 程序
物理地址存储单元在内存中的实际编号，硬件访问使用的最终地址硬件 / 内存

程序执行时 CPU 使用的是逻辑地址，必须通过地址转换机制（如 MMU / 页表 / 段表）映射到物理地址才能访存。

# 7.8 页和页框有何区别？

概念归属说明
页（Page）进程的逻辑地址空间将进程空间划分为大小相等的逻辑分块
页框 / 帧（Frame）内存的物理地址空间将内存物理空间划分为与页面等大小的物理分块

页是进程视角的 "虚拟块"，页框是内存视角的 "实物槽"。操作系统通过页表建立页号到页框号（块号）的映射，一页对应装入一个页框。

# 7.9 页和段有何区别？

维度分页分段
划分依据物理单位，所有页大小固定相等逻辑单位，按程序自身逻辑关系（函数、数组、栈等）划分，段长可变
地址维度一维（仅需一个地址）二维（段号 + 段内地址）
对用户透明性透明，用户感知不到分页用户可见，程序员可按段组织数据
信息共享与保护不方便（可能跨页）更容易（以逻辑段为单位设置权限）
碎片仅内部碎片（最后一页）产生外部碎片（段长不等）

分页是系统视角的 "一刀切"，分段是程序员视角的 "模块化"。段页式管理则结合了两者优势：先按逻辑分段，再在每段内分页。

# 第八章 虚拟内存 概念性复习题

# 8.1 简单分页与虚拟内存分页有何区别？

维度简单分页虚拟内存分页
加载方式进程的所有页必须全部装入内存后才能开始执行只需装入部分页即可运行，其余页按需调入
内存利用率即使某些页暂时不用，也占用内存空间仅当前需要的页驻留内存，利用率更高
并发度受物理内存总量严格限制可运行总空间大于物理内存的进程或更多并发进程
需要页面置换不需要需要，内存满时需换出旧页腾出空间

简单分页是 "全量装入"，虚拟内存分页是 "按需装入"—— 虚存让进程的逻辑地址空间可以远大于物理内存。

# 8.2 什么是抖动？

抖动（Thrashing）是指系统花在页面换入换出（I/O）上的时间远超执行进程本身的时间。此时 CPU 大部分时间在等待磁盘 I/O，吞吐量急剧下降。

根本原因：多道程序度过高，每个进程分到的页框数少于其工作集所需，导致频繁缺页中断。刚换出的页可能马上又要被换入，形成恶性循环。

解决方案：减少并发进程数，增大各进程的驻留集；或采用工作集模型防止抖动发生。

# 8.3 为何在使用虚拟内存时，局部性原理至关重要？

局部性原理是虚拟内存能否实际可行的理论基石。如果一个程序完全随机访问其地址空间，缺页率将极高，系统必然陷入抖动。

局部性类型含义
时间局部性最近访问的指令 / 数据在不久的将来会再次被访问（循环中的变量、频繁调用的函数）
空间局部性即将访问的地址极可能是刚访问地址的邻近位置（顺序执行指令、遍历数组）

得益于局部性，进程在任一较短时间内实际只访问其全部地址空间的一小部分（工作集），因此只需把这一小部分驻留在内存即可高效执行，其余页面可以留存在磁盘上。

# 8.4 哪些元素是页表项中能找到的典型元素？简单定义每个元素。

字段定义
页框号（Frame Number）该页对应的物理页框编号，即地址映射的目标
存在位 / 有效位（P/A）该页是否当前在物理内存中。1 = 在内存，0 = 缺页（在外存）
修改位 / 脏位（M/D）该页自装入内存后是否被修改过。若修改过，换出时必须写回磁盘；若未修改，可直接丢弃
引用位（Referenced）该页最近是否被访问过。用于页面置换算法选择淘汰页，Clock 算法依赖此位
保护位（Protection）该页的访问权限：只读、读写、可执行等。用于存储保护
高速缓存禁止位是否禁用该页的 CPU cache。映射到 I/O 设备内存区域的页面通常需要禁用

# 8.5 转换检测缓冲区的目的是什么？

转换检测缓冲区（TLB, Translation Lookaside Buffer）是一个硬件高速缓存（通常集成在 MMU 或 CPU 内部），用于加速逻辑地址到物理地址的转换。

地址转换需要查页表（可能多次访存），直接查内存太慢。TLB 缓存最近使用的页表项，命中时几乎零延迟完成地址映射；不命中时才去查页表。TLB 利用的是地址转换中的局部性 —— 程序倾向于反复访问同一批页面。

# 8.6 简单定义两种可供选择的页面读取策略。

策略定义优缺点
请求调页（Demand Paging）进程启动时只装入起始所需的最小页面集，之后每当缺页中断发生时才调入所缺页不会装入无用的页，内存利用率高；但初期频繁缺页，启动偏慢
预调页（Prepaging）进程启动或换入时，一次性装入一批页面（通常基于局部性推测后续需要）减少启动阶段的缺页中断次数；但若预取的页未实际使用则浪费内存和 I/O 带宽

实践中通常结合使用：启动时预调页减少初期缺页，运行期采用请求调页按需加载。

# 8.7 驻留集管理和页面置换策略有何区别？

维度驻留集管理页面置换策略
关注问题每个进程分配多少个页框内存满时淘汰哪个页面
决策时机进程创建时、缺页处理时（是否增长）、需要回收时（是否缩减）每次需要调入新页且无空闲页框时
影响范围影响系统整体的内存分配公平性和多道程序度影响单个进程的缺页率
典型问题固定分配 vs 可变分配FIFO、LRU、Clock、OPT 等算法
关系先有 "住多大"（驻留集），才有 "换谁走"（置换策略）

# 8.8 FIFO 和时钟页面置换算法有何联系？

时钟算法（Clock）本质上是对 FIFO 的改进 —— 给 FIFO 加上了 "第二次机会"：

  • FIFO：按页面调入顺序建立队列，淘汰时选队首页。缺点：可能淘汰出频繁使用的老页面（Belady 异常）。
  • Clock：同样按循环队列组织页面，但增加引用位（R 位）。指针扫描时，若 R=1 则清零并给一次机会（跳过），若 R=0 则淘汰。等同于 FIFO 中跳过最近被访问过的页。

时钟算法实现简单、开销低，性能接近 LRU 而远比 LRU 容易实现，是实际操作系统（如 Linux）常用的近似 LRU 算法。

# 8.9 页缓冲实现什么功能？

页缓冲（Page Buffering）维护两个链表：

缓冲区内容作用
空闲页链表被淘汰后尚未分配给新进程的干净页框缺页时可直接取用，无需等待置换写入完成
已修改页链表被淘汰且内容已被修改的脏页系统在空闲时批量写回磁盘，避免缺页时同步等待写入

简言之，页缓冲让 "着急的缺页中断" 不需要等 "慢的磁盘写入"—— 写入可以延迟到后台完成。

# 8.10 为什么不能把全局置换策略和固定分配策略组合起来？

两个策略在逻辑上互斥：

  • 固定分配：每个进程的驻留集大小固定。新页只能替换自己已占用的页框。
  • 全局置换：置换时可以从所有进程的页框中挑选淘汰目标，一个进程可能交出页框给另一个进程。

如果组合使用，全局置换可能取走某进程的页框，导致该进程驻留集缩小 —— 这就违背了 "固定分配" 的承诺。因此：

有效组合说明
固定分配 + 局部置换各自管理，井水不犯河水
可变分配 + 全局置换按需调整，灵活共享
可变分配 + 局部置换按需调整，但仅换自己的页（较罕见）

# 8.11 驻留集和工作集有何区别？

维度驻留集工作集
定义系统实际分配给该进程的页框集合，即进程当前占用的物理页框进程在某时间窗口 Δ 内实际访问过的页面集合 WS (t, Δ)
谁决定操作系统进程自身的访存行为
动态性相对稳定，由 OS 按策略调整随时间窗口滑动动态变化
关系希望驻留集 ≥ 工作集，否则频繁缺页工作集是应分配多少页框的参考依据

工作集模型的核心思想：监视每个进程的工作集大小，确保其驻留集 ≥ 工作集，从根本上预防抖动。

# 8.12 请求式清除和预约式清除有何区别？

维度请求式清除（Demand Cleaning）预约式清除（Precleaning）
写入时机脏页被淘汰时才写回磁盘脏页在后台提前写回磁盘
缺页延迟淘汰时同步等待写 I/O，缺页处理慢淘汰时页已干净，可直接拿用，缺页处理快
写开销只写真正需要写回的页，不浪费 I/O可能白写 —— 提前写干净的页若在被淘汰前再次被修改，前一次写入就是浪费
批量性逐页写入可成批写入，利用磁盘带宽

实际系统（如 Windows、Linux）通常采用两者结合：大部分时候预约式清除在后台工作，负载高时退化为请求式清除，避免不必要的 I/O 浪费。

# 第九章 单处理器调度 概念性复习题

# 9.1 三种处理器调度类型分别是什么？

处理器调度分为三个层次，各自在不同时间尺度上发挥作用：

调度类型别名功能执行频率
长程调度作业调度 / 高级调度决定哪些作业被允许进入系统（创建进程），控制系统的多道程序度最低（几分钟到几十分钟一次）
中程调度交换调度 / 中级调度在内存与外存之间换入 / 换出进程，调节内存中活跃进程数量，平衡系统负载中等
短程调度CPU 调度 / 低级调度从就绪队列中选择下一个获得 CPU 的进程，是操作系统中最核心、执行最频繁的调度最高（几十毫秒一次）

长程调度控制” 谁能进场”，中程调度控制” 谁留在场上”，短程调度控制” 谁上场”。

# 9.2 在交互式操作系统中，最重要的性能要求是什么？

响应时间 —— 从用户发出请求到系统首次给出响应所需的时间。

交互式系统（如桌面、移动设备）用户随时在等待反馈，响应时间直接影响使用体验。若超过几百毫秒无响应，用户就会感知到” 卡顿”。因此调度策略必须以快速响应为首要目标，必要时可牺牲一定的吞吐量。

相比之下，批处理系统关注吞吐量，实时系统关注截止时间的保证。

# 9.3 周转时间和响应时间有何区别？

维度周转时间（Turnaround Time）响应时间（Response Time）
定义从作业提交到作业完成的总时间从请求提交到系统首次产生响应的时间
包含内容等待时间 + 执行时间 + I/O 时间仅到第一次输出的时间点
衡量对象批处理系统的整体效率交互式系统的即时反馈
典型场景数据批处理任务（跑多久不重要，总时长重要）用户点击按钮到界面变化的间隔

# 9.4 在进程调度中，优先级的数值含义是什么？

通常较小的优先级数值表示较高的优先级（priority 0 > priority 1 > priority 2）。这与生活中的” 第一名 > 第二名” 一致。

例如在 Unix/Linux 系统中，用户态优先级的数值越大优先级越低，内核态 priority 值越小越先被调度。不同系统的约定可能相反（有的系统数值越大优先级越高），但” 数值越小优先级越高” 是更常见的惯例。

# 9.5 抢占式调度与非抢占式调度有何区别？

维度抢占式（Preemptive）非抢占式（Non-Preemptive）
定义高优先级进程可主动打断当前运行进程并夺取 CPU当前进程必须主动放弃 CPU 或运行完成，其他进程才能运行
调度时机新进程到达、中断发生、时间片用完、等 I/O 时仅当进程终止、主动阻塞或自愿让出
响应性高，能及时响应高优先级任务低，高优先级进程可能需等待很久
开销上下文切换开销较大上下文切换开销较小
典型代表SRTF、轮转、反馈调度FCFS、SJF（非抢占版本）
实际应用所有现代通用操作系统早期批处理系统

现代操作系统几乎全部采用抢占式，以满足交互和实时性需求。

# 9.6 什么是 FCFS 调度？有何特点？

先来先服务（FCFS, First-Come-First-Served） 按进程到达就绪队列的先后顺序分配 CPU，先到的进程先运行，直到完成或被阻塞。

维度说明
类型非抢占式
实现简单的 FIFO 就绪队列
优点实现极简单，公平（按到达顺序），无饥饿
缺点平均等待时间长；存在护航效应 —— 长进程在前导致后续多个短进程长时间等待
适用场景批处理、打印队列等

护航效应（Convoy Effect）：一个 CPU 密集型长进程长时间占用 CPU，导致所有 I/O 密集型短进程堆积。

# 9.7 什么是轮转调度？有何特点？

轮转调度（RR, Round-Robin） 为每个进程分配一个固定的时间片，进程用完时间片后被暂停并放入就绪队列末尾，CPU 轮转给下一个进程。

维度说明
类型抢占式
实现就绪队列 + 定时器中断
优点公平共享 CPU，响应时间有上限（n 个进程时 ≤ n× 时间片）
缺点时间片选择敏感 —— 太大退化接近 FCFS，太小造成过多上下文切换开销
适用场景分时系统、通用操作系统

时间片通常设为 10~100ms，需要远大于上下文切换时间（约 0.01~1ms）以控制开销比例。

# 9.8 什么是最短进程优先调度？有何特点？

最短进程优先（SPN / SJF, Shortest Process Next / Shortest Job First） 选择当前就绪队列中预估执行时间最短的进程优先运行。

维度说明
类型非抢占式（传统版本）
优点可证明在非抢占式调度中平均周转时间最短
缺点需要预知进程执行时间（不现实）；长进程可能饥饿 —— 源源不断的短进程使其永远得不到 CPU
适用场景批处理系统（运行时间可预估的作业）

执行时间的预知是实现难点：批处理可让用户预估，交互式进程则靠历史行为启发式推测（如指数平均法）。

# 9.9 什么是最短剩余时间优先调度？与 SPN 有何区别？

最短剩余时间优先（SRTF, Shortest Remaining Time First） 始终选择剩余执行时间最短的进程运行。每当新进程到达时，若其剩余时间比当前运行进程更短，则抢占。

维度SPN（SJF）SRTF
抢占性非抢占抢占式
决策时机仅在上一个进程完成或阻塞时每次新进程到达时都可能重新决策
平均周转时间最短（非抢占类中）在所有调度算法中最短（抢占类中）
实现难度较高（需预知时间）更高（需预知时间 + 抢占开销）

SRTF 可视为 SPN 的抢占版本，理论最优但实际中难以精确预知剩余时间。

# 9.10 什么是最高响应比优先调度？解决了什么问题？

最高响应比优先（HRRN, Highest Response Ratio Next） 根据响应比公式选择进程：

响应比 R = (等待时间 + 服务时间) / 服务时间 = 1 + 等待时间 / 服务时间

维度说明
类型非抢占式
核心思想等待越久（分子越大）或服务越短（分母越小），响应比越高
解决的问题SPN 中长进程饥饿 —— 等待时间增大响应比，保证长进程最终得到服务
优点兼顾公平（等待时间）与效率（服务时间），避免饥饿
缺点每次调度需遍历就绪队列计算响应比，开销高于 FCFS

当等待时间趋于无穷时，响应比也趋于无穷，保证长进程迟早获得 CPU。

# 9.11 什么是反馈调度？为何要设计多级队列？

反馈调度（Feedback / Multilevel Feedback Queue, MLFQ） 采用多级优先级队列，进程根据其行为动态调整所处队列：

队列层级优先级时间片调度方式说明
高优先级队列高短（如 8ms）抢占式（任务可能快速完成，响应好）新进程首先进入最高优先级
中优先级队列中中（如 16ms）抢占式短时间未完成则降级到此
低优先级队列低长（如 32ms）轮转CPU 密集型长进程最终降到此层

核心机制：

  • 新进程进入最高优先级，时间片用完后降级到下一层
  • 进程主动放弃 CPU（等 I/O）则保持在当前层甚至升级
  • CPU 密集型进程逐渐降到底层，I/O 密集型进程停留在高层
优点说明
响应性好短 / 交互进程留在高优先级，快速响应
吞吐量高长进程在低优先级别用长时间片减少切换
自适应无需预知进程行为，系统根据运行历史动态调整

反馈调度被誉为” 最优越、最复杂的 CPU 调度方式”，现代操作系统（Linux、Windows、macOS）的调度器均基于此思想设计，是工程实践中调度算法的集大成者。

# 第十一章 I/O 管理与磁盘调度 概念性复习题

# 11.1 列出并简单定义执行 I/O 的三种技术。

技术定义CPU 参与程度数据传输单位
程序直接控制 I/O（Programmed I/O）CPU 通过反复轮询设备状态寄存器来控制数据传送，每次读 / 写一个字。CPU 发出 I/O 命令后需忙等待（Busy Waiting）直到设备就绪全程参与，忙等待一个字
中断驱动 I/O（Interrupt-Driven I/O）CPU 发出 I/O 命令后转而执行其他进程；设备完成数据传送后向 CPU 发中断信号，CPU 再回来处理后续数据仅在开始和中断处理时参与一个字
直接存储器存取（DMA）专用 DMA 控制器接管总线，直接在设备和内存之间批量传输数据，CPU 仅在传输开始和结束时各干预一次仅开始和结束时参与连续的多个块

三种技术的重要演进关系：CPU 干预频率逐步降低，CPU 与 I/O 的并行程度逐步提高。

# 11.2 逻辑 I/O 和设备 I/O 有何区别？

维度逻辑 I/O设备 I/O
抽象层次高层，面向用户 / 进程低层，面向硬件设备
关注点把设备当作逻辑资源，只关心 "要做什么"（读 / 写 / 打开 / 关闭）处理实际物理设备的操作细节，关心 "怎么做"
所在软件层设备独立性软件层设备驱动程序层
接口形式统一系统调用接口（如 read() 、 write() ）设备特定的寄存器操作、命令序列
示例用户进程调用 read(fd, buf, 100) 读 100 字节驱动程序设置磁盘控制器的柱面号、磁头号、扇区号，然后发出读命令

逻辑 I/O 让上层无需关心物理设备差异（不同品牌磁盘、不同接口协议），实现了设备无关性。设备 I/O 则负责将逻辑请求翻译为特定硬件能理解的命令序列。

# 11.3 面向块的设备和面向流的设备有何区别？各举一个例子。

维度面向块的设备（Block-Oriented）面向流的设备（Stream-Oriented）
数据传输单位固定大小的数据块（block），通常 512B ~ 4KB连续的字符流（字节序列），无固定块边界
是否可寻址可寻址 —— 可以随机访问任意块（通过块号定位）不可寻址 —— 只能按顺序读取 / 写入字符流
典型访问方式随机访问（Random Access）顺序访问（Sequential Access）
能否 seek/lseek支持（可跳到任意位置）不支持
是否使用缓冲区缓存通常使用缓冲区缓存（buffer cache）来加速通常不使用或使用简单的字符缓冲区
典型例子磁盘（HDD/SSD）、磁带键盘（输入字符流）、打印机（输出字符流）、网络接口

磁盘是块设备的典型代表 —— 可以读写任意扇区。键盘则是流设备的典型 —— 按键逐个到来，无法 "跳读第 10 个按键"。

# 11.4 为什么希望用双缓冲而非单缓冲来提高 I/O 的性能？

核心原因：单缓冲下 CPU 与 I/O 无法真正并行工作，双缓冲解除了这个限制。

# 单缓冲的处理时序

单缓冲只有一块缓冲区，操作必须串行：

I/O 填满缓冲区 → CPU 处理缓冲区 → I/O 等待 → I/O 填满下一批 → ...

总处理时间 = max(C, T) + M ，其中：

  • C = CPU 处理一块数据的时间
  • T = I/O 传输一块数据的时间
  • M = 数据在缓冲区和用户内存之间的搬移时间

当 C < T 时，CPU 处理完一块后必须空闲等待 I/O 填充下一块；当 T < C 时，I/O 填满缓冲区后必须等待 CPU 处理完才能继续填 —— 两者无法充分重叠。

# 双缓冲的改进

双缓冲维护两块缓冲区 A 和 B，交替使用：

I/O 填入缓冲区 A → 填满后切换到 B → 同时 CPU 开始处理 A
I/O 填入缓冲区 B → 填满后切换到 A → 同时 CPU 开始处理 B

总处理时间 = max(C+M, T) —— 实际上可由较快的一方决定整体速度，只需 C+M ≈ T 即可接近完全并行。

对比维度单缓冲双缓冲
CPU-I/O 并行度低，串行为主高，可近似完全并行
吞吐量受限于较慢的一方接近理论上限
内存开销一块缓冲区两块缓冲区
适用场景极简单设备几乎所有现代 I/O 系统（磁盘、网络、管道）

现代系统中，管道（pipe）、socket 通信、文件 I/O 普遍采用双缓冲甚至循环缓冲，就是为了最大化 CPU 与 I/O 的并行度。

# 11.5 在磁盘读或写时有哪些延迟因素？

一次完整的磁盘访问时间由三部分组成：

延迟因素符号定义量级（HDD 典型值）是否可软件优化
寻道时间T<sub>s</sub>磁头臂移动到目标柱面所需的时间2~15 ms✅ 可通过磁盘调度优化
旋转延迟T<sub>r</sub>目标扇区旋转到磁头下所需的时间，平均为旋转半圈时间平均 2~4 ms（7200rpm 约 4.17ms）❌ 机械决定
传输时间T<sub>t</sub>实际读写数据的时间，与扇区数和旋转速度有关极短（相对前两者）❌ 机械决定

总访问时间 T<sub>a</sub> = T<sub>s</sub> + T<sub>r</sub> + T<sub>t</sub>

三项之中寻道时间通常最大（几十倍于传输时间），因此磁盘调度策略的核心目标就是减小平均寻道时间。旋转延迟和传输时间由磁盘物理参数（转速、扇区密度）决定，操作系统无法改变。

对于 SSD，由于没有机械部件，不存在寻道时间和旋转延迟，三项简化为一维的 Flash 芯片访问时间。

# 11.6 简单定义图 11.7 中描述的磁盘调度策略。

图 11.7 对比了几种典型磁盘调度算法对同一组请求序列的磁头移动轨迹。以下按出现顺序定义：

调度策略定义特点
FIFO（先来先服务）按 I/O 请求到达的顺序依次处理公平但未优化寻道时间，磁头移动量可能很大。不优化平均寻道长度
SSTF（最短服务时间优先）每次选择离当前磁头位置最近的请求处理类似贪心算法，通常寻道效果优于 FIFO；但可能导致饥饿 —— 远离磁头的请求长期得不到服务
SCAN（扫描 / 电梯算法）磁头沿一个方向持续移动，依次处理途中所有请求；到达磁盘一端后反向继续处理改善了 SSTF 的公平性（两个极端的请求最终也会被服务），但中间磁道得到服务的频率是两端的两倍
C-SCAN（循环扫描）类似 SCAN，但到达磁盘一端后不是反向扫描，而是迅速回到另一端重新开始同向扫描消除了 SCAN 的 "中间磁道特权"，各磁道等待时间更均匀；但回程不做服务，浪费了一次移动

四种策略的对比总结：

维度FIFOSSTFSCANC-SCAN
优化寻道❌✅（贪心）✅✅
公平性✅ 按到达顺序❌ 可能饥饿中等（中间磁道更优）✅ 各磁道均匀
实现复杂度极简中中中

实际系统中，Linux 默认使用完全公平队列（CFQ） 或 Deadline 调度器（保证请求的延迟上界），它们都是基于 SCAN/C-SCAN 思想的变体。

# 11.7 简单定义 7 个 RAID 级别。

RAID（Redundant Array of Independent Disks，独立磁盘冗余阵列）通过将多块磁盘组合起来，在性能、可靠性和成本之间取得不同的平衡。

RAID 级别名称原理最少磁盘数容错能力有效容量特点
RAID 0条带化（Striping）数据分条存储在多块磁盘上，并行读写2无，一块坏全坏N × 单盘性能最高，但无冗余
RAID 1镜像（Mirroring）每块数据盘有一块镜像盘，数据完全复制2可容忍 1 块故障50%（N/2）读性能好（可并行），可靠性高，成本高
RAID 2位级条带 + 汉明码以位为单位条带化，使用汉明纠错码3 + 纠错盘1 块故障取决于纠错盘数量实际已淘汰（现代磁盘自带 ECC）
RAID 3字节级条带 + 专用奇偶校验盘数据按字节条带化，一块专用盘存储奇偶校验31 块故障N-1专用校验盘成为写瓶颈
RAID 4块级条带 + 专用奇偶校验盘与 RAID 3 类似但按块（而非字节）条带化31 块故障N-1解决了读粒度问题，但写校验盘仍是瓶颈
RAID 5块级条带 + 分布式奇偶校验校验信息均匀分布在所有磁盘上，无专用校验盘31 块故障N-1消除写校验盘瓶颈，最常用的 RAID 级别之一
RAID 6块级条带 + 双分布式奇偶校验使用两种不同的校验算法，校验信息分布存储42 块同时故障N-2比 RAID 5 多一层保护，适合大容量磁盘阵列

常用级别对比速查：

级别关键特征典型应用
RAID 0纯性能，无保护视频编辑、临时缓存
RAID 1纯冗余系统盘、数据库日志
RAID 5性能 + 冗余，性价比高文件服务器、Web 服务器
RAID 6更强冗余大容量 NAS、归档存储
RAID 10（1+0）条带化 + 镜像（先做 RAID 1 再做 RAID 0）高性能数据库

# 11.8 典型的磁盘扇区大小是多少？

传统磁盘扇区大小为 512 字节，这是从早期磁盘沿用至今的行业标准。

目前出现了两种变化趋势：

扇区大小场景说明
512 字节经典 HDD、旧系统沿用几十年的标准，兼容性最好
4KB（4096 字节）现代大容量 HDD（高级格式化 / Advanced Format）、多数 SSD 的物理页更大扇区提高存储密度（减少 ECC 和间隙的占比），但操作系统需支持

现代操作系统（Linux、Windows 7+、macOS）已经普遍兼容 4KB 扇区，但为了兼容老旧软件，很多 4KB 物理扇区的磁盘仍提供 512 字节逻辑扇区模拟（512e，即 512-byte Emulation）。
