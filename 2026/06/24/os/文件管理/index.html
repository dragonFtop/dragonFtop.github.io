# 文件管理

# 1. Why / 为何需要文件管理

# 1.1 产生背景

程序运行时所需的数据通常先存放在主存中，但主存存在两点固有局限：

  • 易失性：一旦断电或进程结束，其中的数据便不复存在，无法长期保留
  • 容量有限且成本较高：难以容纳用户不断增长的海量数据

因此，必须把需要长期保留的数据存放到磁盘等外存上，以获得稳定、持久的保存。

然而，若让用户直接面对外存，问题同样不容忽视。外存按柱面、磁道、扇区等物理结构组织，如果缺少统一的管理机制，用户就不得不自行记录每段数据所处的物理位置、手动管理空闲空间，既繁琐又极易出错；不同用户或进程之间也难以安全地共享同一份数据。

为了屏蔽这些物理细节，同时满足数据长期保存与共享的需要，操作系统引入了文件与文件管理机制：把外存上一组相关的数据抽象成一个 "文件"，用户只需通过文件名即可访问，而不必关心它在磁盘上的具体位置。

# 1.2 文件管理的目标与基本需求

按照 Stallings 在《操作系统 —— 精髓与设计原理》中的归纳，文件本身通常应具备三项理想性质：

  • 长期存在：文件保存在外存上，不会随用户退出登录而消失
  • 可被共享：文件拥有名字并可设置访问权限，从而支持受控的共享
  • 具有结构：文件可以按层次或更复杂的方式加以组织，以反映彼此之间的关系

围绕这些性质，一个文件管理系统通常需要达成以下几方面的目标：

  1. 满足用户的数据管理需求：支持数据的存储，以及创建、读取、写入、修改等基本操作
  2. 尽可能保证文件中数据的有效性，使用户存取到的是正确、完整的数据
  3. 优化性能：兼顾系统整体的吞吐量与单个用户的响应时间
  4. 为多种类型的存储设备提供 I/O 支持，屏蔽磁盘、磁带等设备之间的差异
  5. 尽量减少数据丢失或被破坏的可能，例如借助校验、备份等手段
  6. 向用户进程提供一组标准化的 I/O 接口，使数据的访问方式保持一致
  7. 在多用户系统中支持多个用户安全地并发访问

从用户使用的角度看，一个可用的文件系统至少应当具备以下功能需求：

  • 用户可以创建、删除、打开、关闭、读写以及修改文件内容
  • 可以在文件之间进行数据移动
  • 可以通过文件名访问用户文件（按名存取）
  • 可以采取适当方式对文件组织结构进行改动
  • 可以在受控情况下访问其他用户的文件
  • 可以控制哪些对其他用户的文件访问操作是合法的
  • 可以对文件进行备份，并且在数据发生错误或损坏时对文件进行修复

# 1.3 文件管理的地位与意义

文件是操作系统用于应用管理和访问数据的抽象，需要满足持久存储、共享访问、结构化组织的功能。文件系统是操作系统中用户接触最直接、使用最频繁的部分。

它最显著的特点是 **"按名存取"**—— 只要给出文件的符号名，系统就能据此找到并存取外存中对应的信息，而把物理地址、空间分配等复杂细节隐藏在背后。从这个意义上说，文件系统在用户所见的逻辑视图与磁盘上的物理存储之间架起了一座桥梁。

# 1.4 文件管理与系统安全

由于文件是数据在系统中长期驻留的主要形式，文件管理也就成为系统安全中不可忽视的一环。数据的机密性、完整性与可用性，最终大多要落到对文件的保护上：哪些主体能够读取、修改或执行某个文件，能否把权限授予他人，都依赖文件系统所提供的存取控制机制来承载。可以说，操作系统的访问控制策略，很大程度上正是通过文件这一层来具体落实的。

# 2. What / 何为文件管理

# 2.1 文件的抽象层次

在文件管理中，数据通常按以下层次结构组织：

  1. 域 (Field)：数据的基本单位，包含一个值，有固定或可变长度。可通过其长度和数据类型（如 ASCII 字符串、二进制数等）来描述。

  2. 记录 (Record)：一组相关域的集合，是应用程序进行处理的基本单元，固定或可变长度。每条记录通常包含一个长度域。

  3. 文件 (File)：一组相似记录的集合，被视为一个整体，通过文件名进行识别和引用，是访问管理的基本单位。文件有唯一的文件名，可被创建或删除。

  4. 数据块 (Block)：辅存中一段连续的物理单位，包含文件的一部分数据，固定或可变长度。它是文件存储空间分配的基本单元，也是系统缓冲的基本单元。

注意：并非所有文件管理系统都呈现这种记录结构。在 UNIX 或类 UNIX 系统上，文件的基本结构是字节流，没有物理域、记录等概念。

# 2.2 文件的典型操作

用户和应用程序希望能够对文件执行多种操作：

操作说明
Retrieve_All（检索全部）检索文件的全部记录，通常等同于顺序处理，适用于批处理应用
Retrieve_One（检索一条）仅要求检索一条记录，面向事务的交互式应用需要这一操作
Retrieve_Next（检索下一条）根据最近检索过的记录，按逻辑顺序检索下一条记录
Retrieve_Previous（检索前一条）检索当前访问记录前面的一条记录
Insert_One（插入一条）在文件中插入一条新记录，为保持顺序需插入到适当位置
Delete_One（删除一条）删除一条已存记录，可能需要更新某些链接或数据结构
Update_One（更新一条）检索一条记录，更新一个或多个域，并把更改后的记录写回文件
Retrieve_Few（检索部分）检索满足某些特定规则的所有记录

# 2.3 文件系统调用

文件系统调用是在二级存储上管理数据 / 文件的抽象接口。常见的功能包括：

  • 基于文件和目录的基本数据管理（创建、删除、打开、关闭、读写）
  • 文件元数据和属性的管理：文件所有者、创建 / 修改的时间、访问或修改权限

基本操作：

操作步骤
创建 (create)1. 在外存中找到文件所需空间 2. 创建该文件对应的目录项
删除 (delete)1. 找到文件名对应的目录项 2. 回收文件占用的磁盘块 3. 删除文件对应的目录项
读 (read)根据目录项找到文件在外存中的位置，读取数据
写 (write)将数据写入文件对应的磁盘块
打开 (open)1. 找到文件名对应的目录项 2. 将目录项复制到内存的 "打开文件表" 中
关闭 (close)递减文件打开计数，当计数为 0 时从内存的打开文件表中移除该文件表项

# 3. 文件目录

考点：目录概念、目录结构、访问控制；多级目录下绝对 / 相对路径解析规则

目录是操作系统对用户文件管理功能的抽象，它本身也是一种文件（存储文件记录的特殊文件），可被各种文件管理例程访问。

目录包含的信息：

类别具体信息
基本信息文件名、文件类型、文件组织方式
位置信息卷、起始地址、文件大小、分配大小
访问控制信息创建者身份、所有者、访问权限设置
使用信息创建时间、最后一次读 / 修改的日期、最后一次修改的用户身份、文件使用状态（是否加锁、是否在内存中被修改等）

目录支持的操作：查找、创建文件、删除文件、显示目录、修改目录。

# 3.1 目录结构的演进

  1. 单级目录结构：所有用户文件在一个目录下，存在用户间文件名冲突问题，不便于管理。

  2. 两级目录结构：主文件目录 (MFD) + 用户文件目录 (UFD)，每个用户有独立目录，消除了用户之间的文件名冲突，但是用户需要解决自己的文件名冲突。

  3. 多级（树形）目录结构：有多级目录组织，每个用户都拥有一个这样的目录结构，只需解决同一目录下的文件名冲突。当代操作系统均采用此结构，层次清晰，但不便于文件共享。

# 3.2 绝对路径与相对路径 ★考点

绝对路径：从根目录出发的完整路径名，所有路径都从主目录（根目录）开始。根目录名是隐含的。

  • 例： /home/user/docs/report.txt
  • 解析过程：从根 / 开始 → 查找 home 目录项 → 查找 user 目录项 → 查找 docs 目录项 → 找到 report.txt

相对路径：相对于当前工作目录 (working directory) 的路径名。每个交互用户或进程总有一个当前路径与之关联。

  • 例：当前工作目录为 /home/user ，要访问 docs/report.txt
  • 解析过程：从当前工作目录 /home/user 开始 → 查找 docs 目录项 → 找到 report.txt

# 3.3 文件控制块 (FCB) 与索引节点 (i 节点)

文件控制块（FCB） 是操作系统为每个文件建立的数据结构，包含文件的各种属性信息。

索引节点（i 节点） 是对 FCB 的优化：将文件名和文件描述信息分离，仅保留文件名在目录项中，其余描述信息存入 i 节点。这样目录项大小减小，检索速度提高。

在 UNIX 系统中，目录项只包含文件名和索引节点号，而文件的所有属性信息（所有者、权限、时间戳、大小、块指针等）都存储在索引节点中。

# 3.4 目录结构的实现方法

方法说明
线性列表目录项按顺序排列，实现简单但查找效率低，需要线性扫描
哈希表使用哈希函数将文件名映射到哈希表的位置，可实现 O (1) 的查找效率，但需处理哈希冲突
B 树使用 B 树或 B+ 树组织目录项，支持高效查找、插入和删除，且保持有序。现代文件系统（如 ext4、NTFS）广泛采用

# 4. 文件的逻辑结构

文件的逻辑结构是从用户角度看到的文件组织形式，它独立于文件在磁盘上的物理存储方式。

# 4.1 无结构文件（流式文件）

由一系列二进制数据流组成，如 .txt 文件中的字符数据。源程序、可执行文件等通常为无结构文件。UNIX 系统中的文件基本结构就是字节流。

# 4.2 有结构文件（记录式文件）

由一组相似的记录组成，每条记录又由若干数据项构成。

# 顺序文件

记录一个接一个顺序排列，可定长或变长，可顺序存储或链式存储。

存储方式定长记录变长记录
顺序存储可随机存取，顺序结构可快速查找无法随机存取
链式存储无法随机存取无法随机存取

按照与关键字顺序的关系分为：串结构（记录顺序与关键字无关）和顺序结构（按关键字排列）。

# 索引文件

为变长记录文件建立一张索引表（定长记录的顺序文件），每条记录对应一个索引项，通过索引可快速定位记录。可以方便地实现随机存取。

# 索引顺序文件

在顺序文件的基础上增加多级索引表，兼顾顺序存取和随机存取。新记录可放入溢出文件，通过链接到对应块末尾来查询，简化更新操作。

# 5. 文件组织方式 ★考点

考点：5 种组织方式的记录结构、访问特性、适配应用场景

文件组织指文件中记录的逻辑结构，它由用户访问记录的方式确定。文件在辅存中的物理组织取决于分块策略和文件分配策略。

在选择文件组织时，有以下重要准则：快速访问、维护简单、易于修改、可靠性、节约存储空间。这些准则可能会自相矛盾，例如为了节约存储空间数据冗余应最小，但另一方面冗余又是提高数据访问速度的一种主要手段（如使用索引）。

# 5.1 堆文件 (Pile)

数据按到达顺序收集，每条记录由一串数据组成，记录可以有不同的域或域相似但顺序不同。每个域都应能自我描述（包含域名和值）。

  • 记录结构：无固定结构，每个域自我描述，自包含域名和值
  • 访问特性：只能穷举查找，性能差
  • 优点：收集数据快、空间使用情况好、易于修改
  • 缺点：只能穷举查找，性能差
  • 适配场景：数据在处理前采集并存储、或数据难以组织的场景（如日志收集）

# 5.2 顺序文件 (Sequential)

每条记录使用固定格式，所有记录具有相同长度，由相同数量、长度固定的域按特定顺序组成。有一个特殊的关键域 (key field)，记录按关键域顺序存储。

  • 记录结构：定长记录，固定格式，按关键域排序
  • 访问特性：顺序存取快，不支持随机访问（除非定长 + 顺序存储）
  • 优点：适合批处理，是唯一可以很容易地存储在磁盘和磁带中的文件组织
  • 缺点：交互式应用性能差，每次更新都要全量查询

优化方式：使用事务日志记录更新操作，再定期与主文件合并，大幅减少每次更新的查询开销。

  • 适配场景：批处理应用（如银行对账单、工资单处理）

# 5.3 索引顺序文件 (Indexed Sequential)

保留顺序文件的关键特征（记录按关键域顺序组织），增加了两个特征：用于支持随机访问的文件索引和溢出 (overflow) 文件。

  • 记录结构：定长记录，按关键域排序，附多级索引表
  • 访问特性：既支持顺序存取，又支持通过索引随机访问单条记录
  • 优点：极大减少访问单条记录的时间，同时保留文件的顺序特性
  • 缺点：溢出文件需要额外维护，需要周期性合并

新记录放入溢出文件，链接到对应块末尾来查询：

  • 适配场景：既需要批处理又需要单条查询的大型数据库（如银行账户系统）

# 5.4 索引文件 (Indexed)

采用多索引结构，成为查找条件的每个域都可能有一个索引。摒弃了顺序性和关键字的概念，只能通过索引来访问记录。记录放置位置无限制，只要至少有一个索引的指针指向即可。

  • 记录结构：变长记录，无顺序约束，多索引指向同一记录
  • 访问特性：只能通过索引访问，支持基于多个域的查找
  • 优点：支持基于多个域的查找，支持变长记录，不需要溢出文件
  • 缺点：任何记录操作后都需修改所有相关索引，索引本身占用空间大

索引文件最大的特点是：记录可以任意放置，不需要溢出文件等机制。但最大的坏处是：任何一个记录进行操作后都需要对所有相关的索引进行修改 —— 增加新记录时必须完善该记录的全部索引，删除记录时也必须删除该记录的全部索引，修改记录时需要更新该记录的所有相关索引。

  • 适配场景：需要多属性查询的信息检索系统（如图书馆书目检索、电商商品搜索）

# 5.5 直接 / 散列文件 (Direct/Hashed)

利用关键域直接计算存储位置，开发直接访问磁盘中任何一个地址已知的块的能力。没有顺序排序的概念。

  • 记录结构：定长记录，通过计算得出存储位置
  • 访问特性：访问速度极快，直接计算即得位置
  • 优点：访问速度极快
  • 缺点：散列存在冲突问题，不适合范围查询

直接文件与散列文件的区别：

  • 直接文件：直接根据关键字计算出记录的存储位置，逻辑上有序排列，适合范围查询。例如：

  • 散列文件：使用散列函数得到存储位置，但存在散列冲突问题，不适合范围查询（因为经过散列处理后顺序不确定）。

  • 适配场景：直接文件适合需要范围查询的快速访问场景；散列文件适合精确匹配、无需排序的快速查找（如哈希索引的 key-value 存储）

# 6. 文件分配方式 ★★重点考点

考点：分区大小设计考量因素；连续分配、链式分配的缺陷与优化；inode 索引结构文件大小计算

文件分配指给定一个文件的所有数据块，决定这些数据块在二级存储中存放的盘块位置。

# 6.1 分区大小设计考量因素 ★考点

文件分配涉及以下三个核心考量：

考量因素说明可选方案
① 分区是否可变创建文件时是否一次性分配最大空间预分配：声明最大大小，一次分配（空间可能浪费，但实现简单）动态分配：按需逐步分配（空间利用率高，但管理复杂）
② 分区尺寸分配单元采用大分区还是小盘块可变大分区：利于局部性，但易产生外部碎片，需压缩整理固定小盘块：灵活、碎片问题轻，但局部性较差
③ 分区是否连续分配给文件的盘块是否必须连续连续：顺序读写快但产生外部碎片不连续（链式 / 索引）：灵活、无外部碎片，但可能增加寻道开销

# 6.2 连续分配 —— 缺陷与优化

文件在磁盘上占有一组连续的块，属于使用大小可变分区的预分配策略。文件分配表中每个文件只需要一个表项：起始块和文件长度。

  • 优点：顺序访问速度快，可同时读入多个块提高 I/O 性能；检索一个块非常容易；文件分配表小
  • 缺陷：
    1. 不方便扩展 —— 文件增长时可能需要整体迁移
    2. 会产生外部碎片
    3. 需要在创建文件时声明文件大小
  • 优化方案：压缩 (Compaction) —— 定期整理磁盘，将碎片空间合并成连续空闲区

# 6.3 链式分配 —— 缺陷与优化

典型情况下基于单个块，链中的每块都包含指向下一块的指针。文件分配表中每个文件只需要一个表项：起始块和文件长度。

  • 优点：方便扩展，任何一个空闲块都可加入链中；无外部碎片，利用率高；文件分配表小
  • 缺陷：
    1. 只支持顺序访问，不支持随机访问 —— 要访问第 N 个块必须从第一个块开始遍历
    2. 局部性差 —— 文件块散布在磁盘各处，顺序处理也需要对磁盘不同部分进行一系列访问
  • 优化方案：周期性合并 (Consolidation) —— 定期将同一文件的块整理到连续区域

# 6.4 索引分配

每个文件在文件分配表中都有一个一级索引，分配给该文件的每个分区在索引中都有一个表项。文件索引保存在一个单独的块中，文件分配表中该文件的表项指向这一块。

  • 优点：支持顺序访问和随机访问；无外部碎片；可基于固定大小的块或大小可变的分区
  • 缺点：索引表本身占用空间；需要不时地进行文件整理

索引表实现方案：

方案说明
链接方案多个索引块链接在一起
多层索引一级索引指向二级索引，依次类推
混合索引综合直接块、一级间接块、二级间接块等（如 UNIX inode）

# 6.5 inode 索引结构与文件大小计算 ★★★核心考点

考点：直接 / 一级 / 二级 / 三级索引支持的最大文件容量分析

UNIX inode 采用混合索引方式，包含直接块指针和间接块指针。

典型 inode 结构（以 ext2 为例，块大小 1KB = 1024B，指针占 4B）：

计算（块大小 1KB = 1024B，指针 4B）：

每个索引块可存放的指针数 = 1024 / 4 = 256 个指针

索引级别可寻址数据块数容量计算容量
直接块 (12 个)1212 × 1KB12 KB
一级间接256256 × 1KB256 KB
二级间接256² = 6553665536 × 1KB64 MB
三级间接256³ = 16,777,21616,777,216 × 1KB16 GB

注意：实际最大文件大小取决于块大小和指针大小：

  • 块越大，每个索引块可存放的指针越多，可寻址的数据块越多
  • 常见的变体：块大小 4KB，每个索引块可放 1024 个指针，三级间接即可达到 TB 级容量

# 7. 文件存储空间管理

存储管理的基本任务：文件分配（决定数据块存放位置）与空闲空间管理（维护可用盘块集合）。除文件分配表 (FAT) 外，还需要磁盘分配表 (DAT)。

方法说明特点
位表法 (Bitmap)使用一个向量，每一位对应于磁盘中的每一块。0 表示空闲块，1 表示已使用块能相对容易地找到一个或一组连续的空闲块，适用于任何文件分配方法。位表通常需要驻留在内存中
链接空闲区使用指向每个空闲区的指针和它们的长度值，将空闲区链接在一起空间开销可以忽略不计，适用于所有的文件分配方法。但使用一段时间后磁盘会出现很多碎片
索引法把空闲空间视为一个文件，并使用索引表能为所有的文件分配方法提供有效的支持
空闲块列表每块都指定一个序号，所有空闲块的序号保存在磁盘的一个保留区中磁盘上用于空闲块列表的空间小于磁盘空间的 1%。可视为下推栈或 FIFO 队列，栈中靠前的数千个元素可保留在内存中，大多数时候能提供零时间的访问
成组链接法目录区中用一个磁盘块作为超级块，记录空闲块信息，兼顾效率和空间开销。是空闲块列表与链式思想的结合UNIX 等系统采用的经典方法

# 8. 文件系统的层次结构 ★考点

考点：识图 —— 文件系统分层设计图（教材图 11.4、图 12.1）

现代文件系统采用分层设计，以屏蔽物理细节并提高可扩展性。典型的文件系统软件架构如下（自顶向下）：

层次职责
访问方法层在应用程序和文件系统以及保存数据的设备之间提供标准接口。不同的访问方法反映了不同的文件结构及访问和处理数据的不同方法（堆、顺序、索引顺序、索引、散列）
逻辑 I/O 层使用户和应用程序能够访问记录。基本文件系统处理的是数据块，而逻辑 I/O 模块处理的是文件记录。提供一种通用的记录 I/O 能力，并维护关于文件的基本数据
基本 I/O 管理程序负责所有文件 I/O 的初始化和终止。维护设备的输入 / 输出、调度和文件状态。参与调度对磁盘和磁带的访问。I/O 缓冲区的指定和辅存的分配也在这一层实现
基本文件系统（物理 I/O 层）这是与计算机系统外部环境的基本接口。处理在磁盘间或磁带系统间交换的数据块，关注的是这些块在辅存和内存缓冲区中的位置，而非数据的内容或所涉及的文件结构
设备驱动程序层直接与外围设备（或它们的控制器或通道）通信。负责启动设备上的 I/O 操作，处理 I/O 请求的完成。对于文件操作，典型的控制设备是磁盘和磁带设备

该分层结构实现了逻辑与物理分离，屏蔽了底层硬件细节。Linux 等系统中还引入了虚拟文件系统层 (VFS)，提供统一的抽象接口，屏蔽不同文件系统的差异；以及缓冲区高速缓存层 (Buffer Cache)，用于缓存磁盘块数据，减少磁盘 I/O 次数。

# 9. 文件共享

在多用户系统中，几乎总是要求允许文件在多个用户间共享。这时就会产生两个问题：访问权限和对同时访问的管理。

# 9.1 共享的实现方式

方式说明优点缺点
硬链接（基于索引节点）多个目录项直接指向同一个 i 节点删除无关性：删一个链接不影响其他不能跨文件系统
软链接（符号链接）存储目标文件的路径（类似 Windows 快捷方式）可以跨文件系统原文件删除后，软链接失效

# 9.2 同时访问

若允许多个用户追加或更新一个文件，则操作系统或文件管理系统必须强加一些规范。一种蛮力方法是在用户修改文件时，允许用户对整个文件加锁。较好的控制粒度是在修改时对单个记录加锁。实际上，这正是读者 / 写者问题。在设计共享访问能力时，必须解决互斥问题和死锁问题。

# 10. 文件保护与访问控制 ★考点

考点：Unix 文件访问权限编码规则

文件保护是指防止文件被未经授权的访问、修改或破坏。

方式说明特点
口令保护访问文件时需要输入口令简单，但口令存储在系统中，安全性一般
加密保护对文件内容加密存储保密性强，不需要在系统中存储密码，但编解码需要时间
访问控制在 FCB 中增加访问控制表 (ACL)，记录各用户可执行的操作灵活精确，是最常用的方式

# 10.1 访问权限的层次结构

权限等级说明
无 (None)用户甚至不知道文件是否存在，更不必说访问它。不允许用户读包含该文件的用户目录
知道 (Knowledge)用户可确定文件是否存在并确定其所有者。用户可向所有者请求更多的访问权限
执行 (Execution)用户可加载并执行一个程序，但不能复制它。私有程序通常具有这种访问限制
读 (Reading)用户能以任何目的读文件，包括复制和执行。有些系统还可区分浏览和复制
追加 (Appending)用户可给文件添加数据，通常只能在末尾追加，但不能修改或删除文件的任何内容
更新 (Updating)用户可修改、删除和增加文件中的数据。包括最初写文件、完全重写或部分重写
改变保护 (Changing Protection)用户可改变已授给其他用户的访问权限。通常只有文件的所有者才具有这一权力
删除 (Deletion)用户可从文件系统中删除该文件

# 10.2 访问权限的授予对象

  • 特定用户 (Specific User)：由用户 ID 指定的单个用户
  • 用户组 (User Groups)：非单独定义的一组用户。系统必须能通过某种方式了解用户组的所有成员
  • 全部 (All)：有权访问该系统的所有用户（公共文件）

# 10.3 UNIX 权限编码规则 ★考点

UNIX 权限模式采用 r / w / x 三种权限，分为所有者 (owner)、组 (group)、公众 (public) 三级，通过八进制数字编码。

编码规则：

权限符号权值含义
读r4可以读取文件内容或列出目录
写w2可以修改文件内容或在目录中创建 / 删除文件
执行x1可以执行文件或进入目录

每级用户的权限 = 权值之和，范围为 0 ~ 7：

八进制二进制权限含义
0000---无权限
1001--x仅执行
2010-w-仅写入
3011-wx写入 + 执行
4100r--只读
5101r-x读 + 执行
6110rw-读 + 写
7111rwx全部权限

常见权限示例：

权限码含义
755所有者 rwx (7)，组 r-x (5)，公众 r-x (5)。常用于可执行文件和目录
644所有者 rw- (6)，组 r-- (4)，公众 r-- (4)。常用于普通数据文件
744所有者 rwx (7)，组 r-- (4)，公众 r-- (4)。所有者可执行，他人只读
600所有者 rw- (6)，组 --- (0)，公众 --- (0)。仅所有者可读写，常用于私密文件
777所有人 rwx。权限最开放，安全风险最高

# 11. 磁盘管理

# 11.1 磁盘物理结构

  • 盘面 (Platter)：磁盘由多个盘面组成，每个盘面两面都可存储数据
  • 磁道 (Track)：每个盘面上的同心圆称为磁道
  • 扇区 (Sector)：每条磁道被划分为若干扇区，是磁盘读写的最小单位
  • 柱面 (Cylinder)：所有盘面上相同半径的磁道组成一个柱面

磁盘物理地址：(柱面号，盘面号，扇区号)

# 11.2 磁盘访问时间

磁盘 I/O 的总时间由三部分组成：

时间分量说明
寻道时间 (Seek Time)磁头移动到目标磁道所需的时间。是影响磁盘性能的最主要因素
旋转延迟 (Rotational Latency)等待目标扇区旋转到磁头下方的时间。平均为磁盘旋转半周的时间
传输时间 (Transfer Time)实际读写数据的时间。取决于磁盘转速和数据量大小

总访问时间 = 寻道时间 + 旋转延迟 + 传输时间

# 11.3 磁盘调度算法

磁盘调度算法的目标是减少平均寻道时间，提高磁盘 I/O 性能。

算法说明优点缺点
FCFS (先来先服务)按请求到达顺序处理公平，实现简单寻道时间长，效率低
SSTF (最短寻道时间优先)总是选择离当前磁头位置最近的请求平均寻道时间短可能导致饥饿（某些请求长期得不到服务）
SCAN (扫描 / 电梯算法)磁头沿一个方向移动，处理沿途所有请求，到达端点后反向公平，无饥饿问题中间区域访问频率高，两端低
C-SCAN (循环扫描)磁头沿一个方向移动处理请求，到达端点后立即跳回另一端（不处理回程请求）对所有位置的请求更公平跳转需要时间
LOOK / C-LOOKSCAN/C-SCAN 的优化，只走到最远的请求就返回（不到端点）减少不必要的移动—

# 11.4 减小磁盘延迟时间的方法

  • 磁盘调度：通过优化 I/O 请求的处理顺序，减少平均寻道时间
  • 高速缓存：在内存中缓存磁盘块，减少磁盘访问次数
  • 提前读 (Read-ahead)：预读后续可能需要的磁盘块
  • 优化文件物理分布：尽量将文件数据连续存放，减少寻道和旋转延迟
  • 交替编号：磁头读取一块内容后需要一小段时间处理，通过交替编号让逻辑相邻的扇区在物理上间隔排列
  • 柱面号在盘面号之前：减少磁头移动消耗的时间
  • 错位命名：相邻盘面的同名扇区错位排列
  • 使用 RAID：通过并行访问多个磁盘提高性能和可靠性

# 11.5 磁盘初始化与管理

  • 低级格式化（物理分区）：划分磁道和扇区
  • 高级格式化：建立文件系统
  • 引导块：ROM 不可修改，只存放很小的 "自举装入程序"。完整引导程序存放在磁盘的引导块中
  • 坏块管理：在 FAT 表上标明坏块（坏块对操作系统透明，文件系统会自动避开）
