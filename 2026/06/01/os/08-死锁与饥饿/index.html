# 死锁与饥饿

# 死锁

  • 死锁：两个或两个以上的进程在执行过程中，由于竞争资源或者由于彼此通信而造成的一种阻塞的现象。

# 可重用资源

  • 每次只能被一个进程使用
  • 被该进程使用完后可以再被其他进程使用
  • 比如：CPU，内存，打印机
  • 死锁发生：每个进程试图占据资源，又申请某个资源，而该资源被其他进程占有

# 可消耗资源

  • 资源可以由进程产生或者消耗掉
  • 产生的资源数量没有限制
  • 被使用后，资源不复存在
  • 比如：中断、信号、消息等
  • 在通信场景中，可消耗资源也可能引发死锁（如进程相互等待对方发送消息）

# 死锁产生的原因

  • 资源竞争
  • 进程推进顺序不当

  • 具体原因可分为四个：
    出现死锁的充要条件：
  • Mutual Exclusion（互斥） ：一次只能有一个进程使用资源
  • Hold and Wait（占有且等待） ：进程持有资源的同时还在等待其他资源
  • No Preemption（非抢占） ：已分配给进程的资源不能被强行抢占
  • Circular Wait（循环等待） ：存在进程资源的环形等待链

# 应对死锁的方法

  1. 鸵鸟算法（无视死锁）
    “无为而治”，忽略死锁，当死锁发生时，更多的是采用手动解决。一般应用于死锁发生极少的场景中。
  2. 死锁预防
    消除死锁的必要条件之一，使得死锁无法发生。
    • 消除互斥：将独占资源改造为共享资源（如通过 SPOOLing 技术将打印机改造为可共享的虚拟设备）。
    • 消除占有且等待：一次性分配进程所需全部资源，该进程直接运行到结束，所有进程均如此。或在请求新资源前释放已占有的全部资源。
    • 消除非抢占：允许操作系统强制收回进程已占有的资源，分配给其他需要该资源的进程。
    • 防止环形等待：定义资源的线性访问顺序，进程按编号递增顺序请求资源，防止形成环形等待链。
  3. 死锁避免
    在资源分配时，判断分配后是否会发生死锁来决定是否这样分配。是一种动态检测的算法，需要提前知道所有的进程未来对需要的需求，need 矩阵。
    该类算法中使用的数据结构：资源矩阵、可用资源矩阵、资源需求矩阵、资源分配矩阵。
    • 进程启动拒绝算法：启动进程时，先检查该进程是否会因资源不足而产生死锁，如无则启动，否则拒绝启动。

    • 进程资源请求拒绝算法（银行家算法）：如果一个进程的增加的资源请求会导致死锁，则拒绝该请求。
  4. 死锁检测、死锁消除
    • 死锁检测：使用状态矩阵，判断当前并发进程是否存在死锁。
    • 死锁消除：1. 撤销死锁进程、2. 回滚到某个检查点再重启、3. 逐个撤销进程、4. 逐个抢占资源、5. 耗费处理器时间最少、6. 输出量最少、7. 已分配资源最少、8. 最低优先级。
    • 死锁检测、死锁消除的问题：死锁检测算法发现的死锁进程不一定会死锁、时间复杂度高、算法自身存在死锁的可能。

# 综合死锁解决策略

针对不同计算机资源，使用不同的策略。

  1. 对计算机资源分类：
    • 可交换空间
    • 进程资源
    • 主存
    • 内部资源
  2. 针对不同资源的策略
    • 可交换空间：一次性请求所有需要的资源
    • 进程资源：死锁避免、找资源排序
    • 主存：抢占式
    • 内部资源：资源排序

# 经典问题 - 哲学家就餐

哲学家就餐问题，也叫死锁问题，是计算机科学中一个经典的问题。
题目要求：

  • 要求一个人能用两把叉子吃饭，所以不允许死锁出现
  • 只有 5 把叉子，但是有 5 个人，所以存在互斥
  • 一个人要用两把叉子吃饭，但是叉子被别人拿走了，所以存在占有且等待
  • 哲学家不可能一把叉子交叉使用，也就是你一口我一口这种方式，必须是一个人吃完后，叉子才给别人。
  • 按照顺序来说，第一个哲学家先吃，第二个和第五个哲学家让第一个哲学家先吃。

综合上述的题目要求，可以使用死锁避免的两种方法：

  1. 进程启动拒绝：最多允许 4 个哲学家同时尝试进餐（保证至少 1 人可以拿到两把叉子，因为 5 把叉子最多凑出 2 对半）
  2. 进程资源请求拒绝算法：哲学家每次拿筷子时判断是否会造成死锁，如果存在死锁，则该哲学家不能使用筷子吃饭。

# 银行家算法详解

银行家算法是死锁避免的经典算法，用于动态判断是否应该满足某个进程的资源请求。

# 数据结构

矩阵 / 向量含义维度
Max （最大需求矩阵）进程对各类资源的最大需求n (进程数) × m (资源数)
Allocation （已分配矩阵）进程当前已分配的资源n × m
Need （需求矩阵）进程尚需的资源 = Max - Allocationn × m
Available （可用资源向量）系统当前可用的各类资源m

# 算法步骤

  1. 检查此次申请是否超过了之前声明的最大需求数：若 Request > Need，则拒绝（进程试图申请超过其声明的最大需求）
  2. 检查此时系统剩余的可用资源是否还能满足这次请求：若 Request > Available，则让进程等待
  3. 试探着分配：系统试探性地分配资源，修改 Available、Allocation、Need
  4. 用安全性算法检查此次分配是否会导致系统进入不安全状态：若安全，则正式分配；否则撤销试探分配，让进程等待

# 死锁的解除

检测到死锁后，必须采取措施解除：

方法说明代价
资源剥夺法挂起某些死锁进程，抢占其资源分给其他死锁进程。可能需要回滚被挂起的进程需要保存 / 恢复进程状态，可能导致部分工作丢失
撤销进程法强制撤销部分甚至全部死锁进程，剥夺其资源被撤销进程的所有工作丢失，代价最大
进程回退法让一个或多个死锁进程回退到足以避免死锁的检查点需要系统支持检查点 / 回滚机制，实现复杂

# 选择 "牺牲者" 的考虑因素

解除死锁时需要选择撤销 / 回退的进程，考虑以下因素：

  • 优先级：优先保持高优先级进程
  • 已运行时间：保护运行时间更长的进程
  • 已使用资源：保护使用资源更多的进程
  • 尚需资源：优先撤销尚需资源多的进程
  • 批处理 vs 交互式：批处理进程比交互式进程更适合作为牺牲者
  • 输出量：保护已经产生输出的进程
  • 被撤销次数：避免反复撤销同一进程
