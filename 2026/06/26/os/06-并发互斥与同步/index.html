英文术语中文翻译定义补充注解
Atomic operation原子操作不可中断、要么全成要么全败的指令序列。保证要么全部执行，要么完全不执行，对外无可见的中间状态。是实现锁、同步的底层基础，比如 CPU 的 test-and-set 指令。
Critical section临界区进程中访问共享资源的一段代码。当另一个进程处于对应代码段时，本代码段不得执行。临界区是并发中需要保护的” 危险区域”，必须通过同步机制保证同一时间只有一个进程进入。
Deadlock死锁两个或多个进程无法继续推进的状态，原因是每个进程都在等待其他进程执行某个操作（如释放资源）。死锁的四个必要条件：互斥、请求与保持、不可剥夺、循环等待。破坏任意一个即可避免死锁。
Livelock活锁进程一直在运行但状态不变、不执行任何有效工作的状态。和死锁的区别：死锁是进程完全阻塞，活锁是进程一直在运行但做无用功。
Mutual exclusion互斥当一个进程处于访问共享资源的临界区时，其他任何进程都不得进入访问这些共享资源的临界区。是临界区的核心约束条件，是解决竞态条件的基础原则。
Race condition竞态条件（竞态）多个线程或进程对共享数据进行读写操作，且最终结果依赖于它们执行的相对时序。并发编程的核心 bug 来源，只有通过同步机制（原子操作、锁、信号量等）才能彻底避免。
Starvation饥饿一个可运行的进程被调度器无限期忽略的状态：尽管该进程具备执行条件，但永远不会被调度执行。常见原因：调度算法优先级不合理、资源分配不公。

核心区别梳理：

  • 死锁 vs 活锁：死锁是完全阻塞，活锁是空转不干活
  • 原子操作 vs 互斥：原子操作是硬件级的不可分割，互斥是软件层面的临界区约束
  • 饥饿 vs 死锁：饥饿是单个进程被无限期忽略，死锁是多个进程互相等待

# 串行与并行

串行执行并发执行
一致性 ：串行执行的程序独享计算机系统资源，只要输入一样，输出结果不会改变非一致性 ：计算机系统资源为并发执行的程序共享，即便输入一样，每次执行的结果可能不同
封闭性 ：串行执行的程序不被打断地执行开放性 ：程序执行一会，停一会，然后再执行一会
连续性 ：一个任务接着一个任务顺序完成中断性 ：一个任务可能在中途被打断，切换到另一个任务

# 互斥与同步

互斥同步
同一时刻只能有一个进程访问临界资源多个进程按一定顺序协调执行
进程访问资源没有固定的顺序要求，谁先请求访问，谁就访问访问临界资源有先后顺序的要求

# 进程交互关系对照表

感知程度关系类型一个进程对另一个进程的影响潜在的控制问题
完全不知情 <br>Processes unaware of each other竞争关系・执行结果独立于其他进程 <br>・执行时序可能受影响・互斥 <br>・死锁（针对可重用资源）<br>・饥饿
间接感知 <br>Processes indirectly aware of each other (e.g., shared object)通过共享实现协作 <br>Cooperation by sharing・执行结果可能依赖于从其他进程获取的信息 <br>・执行时序可能受影响・互斥 <br>・死锁（针对可重用资源）<br>・饥饿 < br>・数据一致性
直接感知 <br>Processes directly aware of each other (have communication primitives available to them)通过通信实现协作 <br>Cooperation by communication・执行结果可能依赖于从其他进程获取的信息 <br>・执行时序可能受影响・死锁（针对可消耗资源）<br>・饥饿

三类场景的核心区别：

  1. 完全不知情 ：进程之间完全不知道对方的存在，仅通过共享资源（如 CPU、磁盘）产生间接竞争，是最基础的并发场景。
  2. 间接感知 ：进程通过共享变量、共享文件等对象产生关联，知道有其他进程在操作同一资源，但不直接发消息。
  3. 直接感知 ：进程通过消息队列、管道、信号等通信原语直接交互，明确知道对方的存在和行为。

核心概念注解：

  • 可重用资源（renewable resource） ：使用后不会被消耗、可重复分配的资源（如 CPU、内存、锁），是竞争 / 共享场景的典型资源。
  • 可消耗资源（consumable resource） ：使用后会被消耗、需要生产者生成的资源（如消息、信号、中断），是通信场景的典型资源。
  • 数据一致性（Data coherence） ：多进程并发读写共享数据时，保证数据最终结果正确、不出现脏读 / 脏写的特性，是共享协作场景的特有问题。
场景核心特征典型问题
竞争关系资源抢占、结果独立互斥、死锁、饥饿
共享协作共享对象、结果依赖互斥、死锁、饥饿、数据一致性
通信协作直接通信、结果依赖死锁（可消耗资源）、饥饿（无互斥需求）

# 临界区同步的 16 字原则详解表

互斥与同步的 P/V 操作要点：

原则条目核心定义核心作用通俗解释关键注意事项
1. 空闲让进当临界区处于空闲状态（无进程访问）时，任何请求进入临界区的进程，都应被允许立即进入保证临界区资源的高效利用，避免资源闲置浪费临界区没人用的时候，谁要进就给谁进，不能故意卡着不让进互斥操作的基础保障，确保资源空闲时能被正常使用
2. 忙则等待当临界区已有进程访问（处于忙碌状态）时，其他请求进入的进程必须等待，不能同时进入保证互斥性，同一时刻仅一个进程进入临界区临界区有人在用的时候，其他想进的进程必须在外面等着，不能硬闯是解决竞态条件的核心约束，直接避免共享资源的并发冲突
3. 有限等待进程请求进入临界区时，等待的时间必须是有限的，不能无限期等待避免饥饿问题，保证所有进程都有机会执行就算临界区一直忙，也不能让某个进程永远等下去，必须给它一个能进去的机会是公平性的核心要求，防止高优先级进程长期霸占资源
4. 让权等待当进程因无法进入临界区而等待时，必须主动释放 CPU 控制权，不能占用 CPU 空转提升 CPU 利用率，避免忙等（Busy Waiting）浪费资源进程进不去临界区的时候，不能占着 CPU 一直循环检查，要把 CPU 让给其他进程是区分忙等算法（如 Peterson 算法）和阻塞算法（如信号量）的关键

# 信号量 (Semaphore)

信号量（Semaphore）是一种用于控制进程访问共享资源的同步原语。信号量的作用是控制对共享资源的访问，并确保共享资源被正确地使用。信号量的基本原理是：当一个进程需要访问共享资源时，它将尝试获取信号量的一个可用许可证。如果获取成功，则该进程可以访问共享资源，并释放信号量的许可证。如果获取失败，则该进程将等待，直到有可用的许可证。信号量的主要作用是确保共享资源被正确地使用，并避免资源被多个进程同时访问。
信号量是一种数据结构。

count 变量为资源计数器，表示资源的剩余数量，也可称为资源许可证数。
queue 变量为等待队列，用于保存等待获取信号量的进程。
semWait() 表示获取信号量的一个可用许可证，如果资源不够，则将进程加入等待队列。
semSignal() 表示释放信号量的一个许可证，如果等待队列非空，则将等待队列中的下一个进程唤醒。

若两个进程之间存在互斥关系，那么需要设置一个信号量，用于控制对共享资源的访问。在一个进程中需要 semWait () 与 semSignal () 成对存在。而两个进程之间的关系为同步关系，则 semWait () 与 semSignal () 分布在两个进程中。

互斥与同步的 P/V 操作要点 ：

规则说明
P/V 成对出现互斥时在同一进程内成对；同步时分布在两个进程中
P 操作顺序同步 P 要放在互斥 P 之前，否则可能死锁
V 操作顺序V 操作不会导致阻塞，可以交换顺序
临界区精简耗时操作不要放在临界区，会降低并发度

# 管程

管程是一种高级同步机制，将共享数据和对数据的操作封装在一起，每次仅允许一个进程在管程内执行。

  • 管程的特征 ：
  • 局部于管程的数据结构只能被局部于管程的过程所访问
  • 一个进程只有通过调用管程内的过程才能进入管程访问共享数据
  • 每次仅允许一个进程在管程内执行某个内部过程

类似于 C++ 中的类：数据放在 private 中，函数放在 public 中。

  • 管程的条件变量 ：用于让进程在管程内等待和唤醒。
  • cwait(condition) ：当前进程释放管程锁，在条件上等待
  • csignal(condition) ：唤醒一个在该条件上等待的进程，被唤醒的进程在调用者释放管程后继续执行

# 阻塞与非阻塞简介

信息的发送和接受可以以阻塞或者非阻塞的方式工作；
阻塞与非阻塞的概念；
通讯的方式：

  • 阻塞发送，阻塞接收：发送与接收进程都需等待；
  • 非阻塞发送，阻塞接收：发送进程不等待，接收进程等待
  • 非阻塞发送，非阻塞接收： 发送和接收进程都不等待

# 阻塞与非阻塞的概念

# 1. 阻塞（Blocking）

阻塞：进程会等待，直到操作完成（如数据成功发送到缓冲区、数据成功从缓冲区读取）后，才会继续执行后续代码。

  • 阻塞发送：发送进程会一直等待，直到数据被完全拷贝到内核缓冲区 / 对端接收完成，才会返回，期间无法处理其他任务。
  • 阻塞接收：接收进程会一直等待，直到有数据到达并被成功读取，才会继续执行，否则会一直挂起。

# 2. 非阻塞（Non-blocking）

非阻塞：进程不会等待操作完成，可以继续执行后续任务，后续通过轮询、回调等方式确认操作是否成功。

  • 非阻塞发送：发送操作调用后立刻返回，数据可能还在发送过程中，发送进程可以立即处理其他工作，无需等待发送完成。
  • 非阻塞接收：接收操作调用后立刻返回，若当前无数据可读，会直接返回 “无数据” 状态，接收进程不会被挂起，可继续执行其他逻辑，后续再尝试读取。

三种通讯组合的核心特点 ：

通讯方式发送进程行为接收进程行为适用场景
阻塞发送 + 阻塞接收等待发送完成等待数据到达简单可靠的同步通讯，如传统 TCP 套接字默认模式
非阻塞发送 + 阻塞接收立即返回，不等待等待数据到达发送端高并发、接收端同步处理的场景
非阻塞发送 + 非阻塞接收立即返回，不等待立即返回，不等待高并发、异步事件驱动的系统，如 Nginx、高性能网络服务
